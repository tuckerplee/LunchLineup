## Legacy User Import

`import-legacy-users.mjs` reads the historical VM106 legacy PHP user export JSON, creates zero-credit tenants, locations, users, preserved legacy password hashes, and RBAC assignments in the Prisma/Postgres schema, and writes the login-method report outside the repo by default. Fresh tenant creation and its exact source-digest zero-wallet/no-ledger provenance commit together under a tenant-slug advisory lock; rerunning an existing tenant never manufactures provenance for an ambiguous older import. It never creates an implicit credit grant; paid work remains blocked until a separate Stripe purchase or admin grant creates the ledger-backed wallet credit. The repeatable `20260716_legacy_unbacked_credit_cleanup.sql` rescan ignores obsolete global completion markers, trusts only exact per-tenant fixed-import provenance, removes the earlier importer's known unbacked 1,000-credit excess only when the locked wallet/ledger history is exact, preserves ledger-backed balances, and aborts on malformed provenance or ambiguous/consumed histories. VM106 is a legacy source label, not the current production target. Run the importer only against an isolated dev/staging database unless production cutover has been explicitly approved. Legacy `super_admin` rows import as tenant `ADMIN` with an `import_note`; create platform admins only through `bootstrap-production-admin.mjs`.

Example:

```bash
DATA_TARGET_ENV=development \
  node scripts/import-legacy-users.mjs /tmp/legacy-users-20260603.json --report /tmp/imported-user-credentials-20260603.csv
```

`DATA_TARGET_ENV` and a valid `DATABASE_URL` are mandatory. A production cutover must use `production-cutover`, run with `NODE_ENV=production`, set `LEGACY_IMPORT_PRODUCTION_CONFIRM=import-legacy-users-production-cutover`, and set `LEGACY_SOURCE_EXPORT_SHA256` to the exact 64-hex SHA-256 of the selected export. The script hashes the export and rejects a mismatch before loading Prisma.

`apply-db-migrations.mjs` requires `MIGRATION_DATABASE_URL` for the database owner/admin and preserves `DATABASE_URL` for the restricted runtime role. Every Prisma, bootstrap, role-provisioning, and webhook-rotation subprocess has an explicit fail-closed deadline. The runner executes the checked-in repository-local Prisma CLI through Node, avoiding shell or `npx` dependency resolution on Windows and CI. Before any Prisma or raw SQL DDL, it invokes `rotate-webhook-endpoint-secrets.mjs --verify-only`; plaintext, malformed, lock-contended, or envelopes outside the configured current/previous keyring fail closed without writes or credential output. It executes ledger-owned `pre_*.sql` transactions before `prisma db push`, then ledger-owned forward SQL transactions after schema synchronization. The private receipt table binds exact path, digest, byte count, phase, source SHA, and applied/baselined mode while a session advisory lock serializes all raw migration runners. Production requires a full `MIGRATION_SOURCE_SHA`; VM217 derives it from the verified candidate release and derives `MIGRATION_BASELINE_SOURCE_SHA` from the authenticated current release. An empty production ledger without that baseline fails unless `MIGRATION_FRESH_DATABASE_CONFIRM=initialize-fresh-production-ledger` explicitly attests a proven fresh database. The inventory skips immutable superseded bootstrap SQL plus schema-superseded legacy username and RBAC migrations, then installs tenant-context helpers, the platform-admin capability, and the corrected forward RBAC owner before dependent migrations. After schema/raw SQL, restricted-role provisioning, and optional production-admin bootstrap all succeed, the helper re-encrypts endpoint and nonterminal delivery envelopes under `WEBHOOK_DELIVERY_ENCRYPTION_KEY_CURRENT` in one serializable transaction. Candidate activation separately requires proof that the retained release preloaded the candidate key for rollback. Ephemeral CI supplies a fixed test-only webhook key; staging and production must receive their protected runtime key.

## Destructive Data Targets

The development Prisma seed, E2E seed, legacy import, and migration wrapper all use `data-target-guard.mjs` before Prisma loads. Seed scopes reject production environment markers and production-like database URLs. Development seed scopes are `test`, `disposable`, or `development`; E2E accepts only `test` or `disposable` and has no production override. Migration contexts are `test`, `disposable`, `development`, `staging`, or `production`; production requires `DATA_TARGET_ENV=production`, `NODE_ENV=production`, plus `MIGRATION_PRODUCTION_CONFIRM=apply-lunchlineup-production-migrations`. The production launch validator enforces all three values, and production admin bootstrap runs only in that context.

## Deploy Rule

Run `verify-deploy-source.ps1` or `verify-deploy-source.sh` before server rollout. A server deploy must match a clean GitHub-pushed SHA and should leave or verify `DEPLOYED_GIT_SHA`. The verifier accepts normal upstream branches and detached GitHub Actions checkouts when `GITHUB_ACTIONS`, `GITHUB_EVENT_NAME=push`, `GITHUB_REF`, and `GITHUB_SHA` prove the pushed branch head.

Run `verify-release-artifacts.mjs` against the CI `release-manifest` artifact before staging or production rollout. App images must be pinned as `<service>:<git-sha>@sha256:<digest>`, every Dockerfile base image and non-app Compose image must include an immutable `@sha256:` digest, and the deployment contract must bind every raw SQL migration path and digest. Candidate launch-proof entries carry source, command, checksum, byte count, unique URI, and fresh timestamps; candidate launch proof must not contain `externalHealth`. Public release identity is established only after mutation by `verify-external-health-release.mjs`, which requires the served `X-LunchLineup-Release` header to equal `RELEASE_SOURCE_SHA` before release pointers advance. `--launch-proof-mode rollback` verifies an older known-good proof's contract, source, and evidence structure without applying the candidate freshness TTL.

Run `validate-production-launch.mjs /path/to/runtime.env` before production deploys. It requires exact 32-byte MFA, webhook, password-reset, and availability-import encryption keys and rejects reused effective key material across payload roles. The bundled production Compose path requires both database URLs to target `postgres:5432/POSTGRES_DB`, so logical backup and PITR cover the authoritative database; the external availability key must remain in managed secrets for the PITR window. VM217 production deploy reruns validation with local secret-file proof before mutation.

The production GitHub environment must define `PRODUCTION_RUNTIME_SECRET_REFERENCE` and `PRODUCTION_RUNTIME_SECRET_VERSION` as variables identifying one immutable AWS Secrets Manager version. The deploy workflow fetches that exact version into a runner-temp file, runs `validate-production-launch.mjs`, records `PRODUCTION_RUNTIME_ENV_PATH`, `COMPOSE_SERVICE_ENV_FILE`, and `PRODUCTION_RUNTIME_ENV_SHA256` in the job environment, rechecks the file hash immediately before deploy, and deletes the temp file after the production deploy job finishes. Before any deploy mutation, `verify-stripe-meter-config.mjs` uses the live Stripe key in that exact runtime env to retrieve `STRIPE_METER_ID`, every configured plan price, and `STRIPE_WEBHOOK_ENDPOINT_ID`. It requires an active live `last` meter with the worker's exact payload mappings, active live metered recurring prices attached to that meter, and an enabled live webhook endpoint subscribed to every handled entitlement event; those values must exactly match `evidence.stripeMeter` in the decoded launch-proof bytes. Normal mutation accepts no `PRODUCTION_DEPLOY_COMMAND`: CI directly invokes `scripts/deploy-vm217-transport.sh` with protected host/user/private-key/pinned-known-host inputs and exact manifest/runtime/proof/SHA arguments. The transport alone forwards `PRODUCTION_API_HEALTH_URL`, `PRODUCTION_WEB_URL`, and `LAUNCH_PROOF_MANIFEST_URI` through its fixed encoded remote environment.

Automatic rollback uses `PRODUCTION_RELEASE_REGISTRY_URI`, not a mutable GitHub rollback-state secret. One protected `production` job resolves `index.json`, downloads immutable `releases/<currentSuccessfulSha>.json`, and validates it before creating `PREVIOUS_DEPLOYMENT_APP_DIR`. After all production input preflights, a dedicated completed workflow step durably arms rollback before the checked-in transport can mutate VM217. Deploy, smoke, conditional candidate publication, and automatic rollback remain inside that already-approved job. Every post-arm outcome is evaluated with `always()`, and the job requires rollback materialization, mutation, and independent proof to succeed whenever any release phase fails.

Production release-registry publication supports S3 only and defaults `RELEASE_REGISTRY_IMMUTABLE_RETENTION_DAYS=35` and `RELEASE_REGISTRY_LIFECYCLE_MAX_RETENTION_DAYS=90`. Before any object write and again before pointer advance, it requires enabled versioning, enabled default COMPLIANCE Object Lock within those bounds, an obsolete-tag-only current-object expiry rule, a bounded noncurrent-version rule, and an unconditional bucket-policy deny for both `s3:DeleteObject` and `s3:DeleteObjectVersion` over the registry prefix. The active release bundle, both signatures, immutable index, and mutable pointer pair are tagged `lunchlineup-release-retention=active`, so a no-deploy interval cannot expire recovery material. Superseded immutable objects receive renewed minimum COMPLIANCE retention and the `obsolete` tag, bounding cleanup without shortening rollback protection. Every immutable put requests SHA-256 and COMPLIANCE retention explicitly; provider `head-object` readback must return a non-null version ID, exact byte count and checksum, no delete marker, and non-premature retention timestamps.

`deploy-vm217-remote.sh` defaults to the production artifact path. Transported runtime bytes remain `root:root` mode `0600`. Before Compose starts, the helper verifies their digest and atomically persists the candidate at a release-SHA/digest-bound `root:lunchlineup` mode-`0640` path behind mode-`0750` directories. The service group is bounded to a resolved non-root local GID, must match the fixed `lunchlineup` service account's primary GID, and fails closed if overridden incorrectly. Compose and staged backup/PITR jobs use that exact path and digest; promotion advances its durable active pointer, while deploy failure or retained rollback restores the matching prior runtime. Every production deploy, retained rollback, backup, and PITR invocation uses the fixed Compose project `lunchlineup` with an explicit candidate project directory, env file, and Compose file. Release SHAs may select immutable source bytes and images but may never become a Compose project name, so persistent volume identity stays stable across releases. Scope-affecting runtime overrides fail before Docker is invoked. Runtime values are never logged or copied into repository/release artifacts. The deploy then pulls only digest-pinned images, starts with `--no-build --pull never`, and requires public identity plus a continuous 15-minute window with no pending or firing critical LunchLineup alert before release pointers advance. Any failed sample resets that window, and the boot-grace-plus-stability deadline bounds the total wait. The canonical production identity is `/opt/lunchlineup/current/DEPLOYED_GIT_SHA`.

For rollback, set `VM217_DEPLOY_OPERATION=rollback`. Before production DDL, CI generates a run-owned clone ID and clone-only environment distinct from the production runtime. The compatibility harness launches its own fixed provider mode inside the immutable clone-provider image, applies the candidate migration entrypoint, independently proves the resulting schema equals the contract-bound candidate Prisma model, fingerprints the canonical public catalog, and runs the retained old release's fixed integration smoke. The raw gate rejects changed historical digests, approves only classifier-proven additive SQL directly, and requires every non-additive migration to have a digest-bound expand/contract record plus that proof. Rollback retains candidate schema, runs the independent live Prisma diff, and keeps migration mode `skip` with SHA-bound confirmation.

`old-release-compatibility-harness.mjs run` accepts no execution adapter. ## Backup And Restore Helpers

`backup.sh` writes encrypted `.sql.zst.gpg` backups and portable `.sha256` sidecars. Production offsite storage must be one non-root `s3://bucket/prefix`; mutable rclone repositories are rejected. Before upload it proves versioning, default COMPLIANCE Object Lock, bounded current/noncurrent lifecycle expiry, an unconditional all-principal deny for `DeleteObject` and `DeleteObjectVersion` on the exact prefix, and an authenticated caller ARN. Each object uses conditional create, explicit SHA-256 and COMPLIANCE retention, then exact version-specific provider readback. Expiry is lifecycle-owned; the script never remotely deletes or prunes. `verify-backup-readiness.sh` requires the matching fresh `offsite_retention_ok`, `offsite_immutable_ok`, and `backup_ok` fields before timers can enable. It snapshots all four installed backup service/timer units and every unit's enabled/active state before mutation; any failure replays exact bytes and runtime state, independently verifies readback, and keeps the snapshot if any restore or confirmation fails. Prefer `BACKUP_ENCRYPTION_KEY_FILE` over inline key material.

Production recurrence is owned by `infrastructure/systemd/lunchlineup-backup.service` and `.timer`. The unit runs the Compose `backup` service only under the `ops` profile, consumes `/run/secrets/backup_key`, forces an offsite target, writes `lunchlineup_backup.prom` into the node-exporter textfile collector, and uses `--pull never` with the last successfully deployed release pointer. Its one-shot wrapper starts a detached container under the fixed `lunchlineup` project, owns the exact returned container ID through wait/log/remove, and proves that ID is absent before reporting completion.

`restore.sh` refuses `latest`, plaintext dumps, missing checksum sidecars by default, and non-empty targets without `RESTORE_ALLOW_NONEMPTY=YES_OVERWRITE`. Production additionally requires an independently signed target pin containing the PostgreSQL `system_identifier` plus a digest-pinned protected descriptor that binds cluster ID, owner host/port/database/user/system identifier, and independently parsed migration URL identity before confirmation or database access. Its confirmation is `restore-production-target:<cluster>:<database>:<target-pin-sha256>:<descriptor-sha256>`, and the script queries `pg_control_system()` and matches the signed identifier before table inspection or mutation. It also requires externally retained signed recovery-adapter and recovery-execution attestations: the fixed workflow signer must bind the exact release/run, immutable source and provider readback, isolated target identity/system identifier, and successful restored-table/app-role outcome. Recomputed but unsigned self-consistent evidence is rejected.

`dr-drill.sh` restores an explicit encrypted backup into an ephemeral Postgres container, runs a sanity query, and retains JSON proof. Public-launch proof is new-file-only and remains absent until bounded container removal plus exact ID/name Docker readbacks confirm absence; cleanup uncertainty exits nonzero and invalidates any proof. The fetch and readback executables must be pinned by a short-lived signed `lunchlineup-signed-recovery-adapter-provenance` attestation retained outside caller evidence. The drill snapshots the attestation, bundle, and both adapters before Cosign, then executes and hashes only those private snapshots; it also snapshots provider output before parse/hash normalization. Launch verification independently verifies both that adapter signature and the signed exact execution/readback/target/outcome/cleanup binding; self-consistent caller JSON cannot authorize arbitrary adapters, forge a successful recovery, or omit container-absence evidence.

```bash
BACKUP_FILE=/tmp/lunchlineup-YYYYMMDDHHMMSS.sql.zst.gpg \
DR_OFFHOST_SOURCE_URI=s3://lunchlineup-prod/db-backups/lunchlineup-YYYYMMDDHHMMSS.sql.zst.gpg \
DR_OFFHOST_SOURCE_VERSION=<immutable-provider-version-id> \
DR_OFFHOST_EXPECTED_SHA256=<64-hex-backup-sha256> \
DR_OFFHOST_FETCH_COMMAND=/usr/local/libexec/lunchlineup-fetch-dr-backup \
DR_OFFHOST_READBACK_COMMAND=/usr/local/libexec/lunchlineup-readback-dr-backup \
DR_RECOVERY_ADAPTER_ATTESTATION_FILE=/secure/recovery-adapter-attestation.json \
DR_RECOVERY_ADAPTER_SIGNATURE_BUNDLE_FILE=/secure/recovery-adapter-attestation.sigstore.json \
DR_RECOVERY_ADAPTER_ATTESTATION_URI=s3://lunchlineup-prod/recovery-adapters/attestation-<digest>.json \
DR_RECOVERY_ADAPTER_SIGNATURE_BUNDLE_URI=s3://lunchlineup-prod/recovery-adapters/attestation-<digest>.sigstore.json \
DR_RECOVERY_ADAPTER_CERTIFICATE_IDENTITY=https://github.com/tuckerplee/LunchLineup/.github/workflows/ci.yml@refs/heads/main \
DR_RECOVERY_ADAPTER_OIDC_ISSUER=https://token.actions.githubusercontent.com \
DR_SOURCE_SHA="$(git rev-parse HEAD)" \
BACKUP_ENCRYPTION_KEY_FILE=/run/secrets/backup_key \
./scripts/dr-drill.sh
```

Launch-proof drills must set `DR_SOURCE_SHA` to the release SHA. Their retained JSON `completed_at`/`checked_at` must exactly equal the outer `drDrill.checkedAt`, `source_sha` must equal the outer evidence `sourceSha`, and `source_readback_verified` must be `true` with the declared version, checksum, retrieved byte count, readback digest, and fetch-adapter digest. Local-only drills may set `DR_REQUIRE_OFFHOST_SOURCE=false`; checksum-free drills require `DR_REQUIRE_CHECKSUM=false` and do not satisfy launch proof.
The default DR Postgres image is digest-pinned. Any `DR_IMAGE` override must also include an immutable `@sha256:` digest.

PITR is a separate physical recovery path. Every encrypted archive upload uses conditional create; explicit immutable versions are enumerated, byte-read back, and retention-verified before success. Provider stat authorization/transport errors are unknown state, never interpreted as absence. Restore requires exact version IDs for `COMPLETE`, `base.tar.gz`, `backup_manifest`, and the named WAL; it rejects missing, multiple, delete-marker, unversioned, or different current versions and downloads with `--version-id`. Every MinIO call remains timeout/TERM/KILL bounded. Run `pitr-verify-storage.sh` before deployment and follow `docs/runbooks/postgres-pitr-recovery.md`.

## Retained Record Expiry Helper

`invoke-retained-record-purge.mjs` is the scheduler-safe wrapper for `POST /api/v1/admin/retention/purge-expired`. It requires an explicit endpoint URL and bearer token file, defaults to `dryRun: true`, takes a lock, and follows stable continuation pages up to `RETENTION_PURGE_MAX_PAGES` (default 20). Proof and Prometheus textfile output include mode, stage, candidate, deleted-record, dormant-session eligible/purged, processed, failed, and skipped counts; every metric carries both `mode` and `stage`, preventing retained-record dry-run success from satisfying automatic application-data execution freshness. The first application-data page applies and reports the bounded global session retention batch plus pending and reconciled deletion-billing barriers; Stripe retry failures are included in `failedTenants` and `failedTenantCount`. Any `failedTenants` or remaining continuation at the safety bound makes the wrapper exit nonzero after isolated attempts complete. The token file must contain the generated retention service token that the API also mounts through `RETENTION_PURGE_SERVICE_TOKEN_FILE`; do not use a normal platform-admin JWT.

Example daily dry-run invocation:

```bash
RETENTION_PURGE_URL=https://lunchlineup.com/api/v1/admin/retention/purge-expired \
RETENTION_PURGE_TOKEN_FILE=/run/secrets/retention_purge_token \
RETENTION_PURGE_PROOF_FILE=/var/lib/lunchlineup/proofs/retention-purge-latest.json \
RETENTION_PURGE_METRICS_FILE=/var/lib/node_exporter/textfile_collector/lunchlineup_retention_purge.prom \
RETENTION_PURGE_LOCK_FILE=/var/lock/lunchlineup-retention-purge.lock \
node scripts/invoke-retained-record-purge.mjs
```

The retention service token is restricted by the API to server-time dry-runs and ignores a caller-supplied `asOf`. Reviewed physical purge execution requires an authenticated platform-admin JWT plus `dryRun: false` and `executeConfirmation: purge-expired-retained-records`; the scheduled service token cannot execute it.

## First Release Registry Bootstrap

An empty registry is not permission to deploy without rollback. Before dispatch, independently retain the secret-free v2 bundle for the release currently serving production. The bundle may contain only the immutable runtime-secret descriptor (provider, reference, VersionId, and SHA-256), never runtime secret bytes.

Run the production workflow manually with:

- `bootstrap_release_registry=true`
- `bootstrap_live_source_sha=<40-character SHA currently served by production>`
- `bootstrap_live_release_bundle_uri=<independent https:// or s3:// retained bundle URI>`
- `bootstrap_confirmation=bootstrap-current-live-release:<same live SHA>`

The protected bootstrap-only dispatch requires exactly one API-health proof at the configured URL and one strict public-HTML proof at the canonical HTTPS root, both serving that exact SHA, then imports the retained bundle with conditional-create semantics; it cannot deploy a candidate. On a later push to main, CI resolves and materializes that baseline, blocks if its authenticated current SHA differs from the independently served live SHA, verifies its rollback contract and retained launch proof, and runs old-release compatibility against the candidate schema clone before mutation. One retained baseline and a completed rollback-arm step keep the centralized rollback job armed for any later deploy-job or smoke-job failure without depending on the mutating command's output stream. After rollback, CI verifies post-deploy proof, API, and strict public HTML, authenticates the then-current registry pointer, permits only the proven candidate SHA or already-restored previous SHA, and uses that readback as the conditional repoint expectation. Candidate runtime bytes exist only in mode-0600 runner-temporary files and are deleted; registry objects never contain them.

## Legacy PHP Initial VM217 Cutover

`initial-vm217-cutover.sh` exists only for the first transition from the legacy PHP service, when no v2 release header or retained v2 rollback bundle can exist. It requires external executable files for snapshot creation, durable proof retrieval, and rollback; all three must live outside this repository, be digest-bound, and be non-symlink files that are not group- or world-writable. Each adapter has an independent finite deadline. A snapshot timeout is reconciled only by valid durable proof, proof fetch retries once before mutation is blocked, and every external rollback result requires pinned-host readback proving exact legacy Apache traffic, no v2 marker/current pointer, and zero v2 project containers. The proof-fetch executable must retrieve fresh JSON from an exact `https://` or `s3://` retained object into a new mode-0600 runner-local file. The attestation binds VM217, `legacySystem=php`, the candidate SHA, snapshot reference and time, durable URI, and exact snapshot, proof-fetch, and rollback executable digests.

The wrapper validates the requested VM217 entry in the supplied pinned `known_hosts`, refuses the initial path when VM217 already has `/opt/lunchlineup/DEPLOYED_GIT_SHA`, passes the identity and pinned-host paths to the external recovery executables, and delegates mutation only to `deploy-vm217-transport.sh`. A successful initial cutover does not seed, relax, or bypass the retained release registry.
