# Lunch and breaks route

## Files

- `README.md`: this route guide.
- `lunch-break-generation-recovery.ts`: durable opaque generation identities keyed by complete mode/location/day/tenant/user/session intent, with per-session fail-closed capacity and payload-free reload recovery.
- `lunch-break-load-ownership.ts`: pure scope claims for rejecting stale refreshes plus exact request-token busy ownership and cleanup for setup, scheduled generation, and manual generation.
- `lunch-break-scope.ts`: monotonic location/day visit epochs plus an atomic active-scope commit guard used to reject stale and A-to-B-to-A planner completions.
- `lunch-break-time.ts`: location-timezone day-window, wall-clock, overnight-shift, and display helpers.
- `page.tsx`: responsive lunch and break planner with bounded reads, keyboard/focus-safe setup controls, idempotent mutation callers, and location/day-owned delayed completion commits.
- `setup-shifts-recovery.ts`: deterministic non-expiring setup identity recovery using cross-tab-safe per-attempt records, opaque session-partitioned capacity, fingerprint-scoped duplicate-submit exclusion, and HTTP error normalization without persisted shift payloads.
- `shift-break-update-recovery.ts`: shift/date/location/tenant/user/session-scoped manual-break recovery, duplicate-submit exclusion, expiry, and stable HTTP error normalization.

## Notes

`page.tsx` requires both `lunch_breaks:read` and `locations:read` plus an explicit tenant location, scopes lunch/break reads, setup-shift writes, row edits, and persisted generation to that location, and converts selected dates and wall-clock times with its IANA timezone. Day reads send explicit UTC bounds and a 200-row page size, then follow opaque cursors to completion; one Strict Mode-safe bootstrap owns the initial sequence, with later sequences owned by explicit scope changes or retry actions. Each scope transition receives a new monotonic visit epoch and synchronously clears prior rows, run banners/previews, planner mode/guide, and derived employee/count state. Setup, scheduled-generation, and manual-generation busy state is owned by an exact request token plus the captured location/day/epoch: a delayed A is not busy in B, and A cleanup cannot clear a newer B owner. Every delayed policy, row-save, setup, scheduled-generation, and manual-generation completion uses the captured location/day/epoch before changing success, error, loading, planner, preview, or banner state. A deferred old A completion remains stale after A-to-B-to-A and cannot claim refresh ownership. Stale request completions are rejected before they can increment the request owner or clear the active day. A failed current-scope read renders an unavailable state with writes disabled until retry succeeds. The setup roster follows the same bounded paging contract before client-side name sorting. Navigation, proxy enforcement, and page bootstrap share the same two-permission prerequisite. Responses are accepted only while their full visit scope matches the active selection, and shift writes remain disabled until that scope has loaded. Manual planning starts empty and requires explicit user-entered rows; it does not seed local dummy staff.

Setup-shift persistence keeps up to eight unresolved identities per opaque tenant/user/session partition. Each identity is a deterministic SHA-256 binding of the complete visible intent and API request, is stored in its own localStorage record to avoid shared read/modify/write loss across tabs, and never rotates merely because 24 hours elapsed or storage disappeared. The v2 manifest is diagnostic compatibility data only. At eight unresolved identities, a different request in the same session fails closed before network submission without evicting an older key; a later login has independent capacity. Employee and shift request payloads remain in memory only. Two tabs and storage-loss retries derive the same key, while the API's tenant-locked semantic marker for new unassigned rows protects shift creation and separate-credit settlement even when clients present different keys. Confirmed success clears only the matching local record; exact repeated intent remains deterministic.

Scheduled and manual generation use a durable map keyed by complete mode/location/day/tenant/user/session plus request semantics. A-to-B-to-A navigation, response loss, reload, and local storage loss therefore recover the original A key instead of a volatile mode slot. The same unchanged intent also retains that key after current 403 entitlement failures or rolled-back 5xx persistence failures, allowing the API to recover after subscription, separate-credit, or database health restoration without a second committed debit. Recovery records contain only opaque fingerprints, keys, and timestamps, capacity is partitioned by active session, and the API's generation request ledger settles persisted generation exactly once. Manual shift-break saves retain their existing 24-hour shift/date/location/tenant/user/session recovery contract. The UI preserves the active-paid-subscription requirement and separately purchased credit cost, surfaces stable 403 entitlement and 409 conflict details, and never changes wallet state locally.

The selected-location catalog is bounded separately from day data. Operators explicitly load each additional location page; selector pages are merged by ID and never auto-drained.
