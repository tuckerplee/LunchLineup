- `manual-shift-schedule.ts`: pure containing-draft selection and complete local-day fallback-window helpers for manual shift creation.
- `page.tsx`: tenant-scoped calendar workspace backed by one v2 board read and aggregate revision-fenced scheduling mutations.
- `publish-attempt.ts`: schedule-and-accepted-preflight-bound publish idempotency attempt helper for stable replay keys and drift rotation.
- `publish-result.ts`: pure delivered, pending, and terminal-failure notification outcome mapping used by the workspace status banner.

- `shift-update-recovery.ts`: payload-bound shift-update attempt persistence and matching-response cleanup across authentication refresh, response loss, and browser reload.
- `print/`: old-style printable schedule route; see `print/README.md`.

## Notes

The calendar loads locations, roster, overlapping schedules, schedule revisions, and full selected-schedule shifts through one generated-client `GET /api/v2/schedule-board` request. It creates schedules through the location collection and persists shift create/update/delete work through `POST /api/v2/schedules/:id/change-sets`; there are no browser-side per-shift v1 writes. Demand replacement and reopening are native revision-fenced v2 operations. Publish, auto-schedule, and break generation also use v2 resource paths while their mature billing/queue subsystems remain private behind the API v2 compatibility boundary. Superseded board responses are discarded, and location-scoped writes remain disabled until the selected location, date, and view have current data.

Persisted timestamps remain UTC ISO instants. First use creates a draft schedule, then requires at least one saved demand window before auto-scheduling. Auto-scheduling a nonblank draft requires a second confirmation before replacement. Auto-schedule polling remains active after a location, date, or view change so the server job can finish, but a solve-generation and captured-scope guard prevents that stale poll from changing the current status or reloading its old calendar scope. Each charged operation keeps one payload-bound `Idempotency-Key` through response loss and authentication refresh. Manual change sets additionally carry the loaded schedule ETag; a stale `412` forces one authoritative board reload instead of issuing conflicting per-row retries. An ambiguous break-generation POST replays the same key, and post-generation reconciliation never repeats the charged POST. Auto-schedule attempt and public job identity are retained in `sessionStorage`; reload resends only an unresolved original request with the same key or resumes the known job status URL, then clears recovery state on terminal completion.

Calendar route access and navigation require `schedules:read`, `shifts:read`, and `locations:read`, matching its unconditional schedule, roster, location, and shift reads. `admin_portal:access` does not bypass missing tenant read permissions. Any failed required read leaves the calendar in an explicit error state; schedule failures are never converted into an empty schedule. Shift editing requires `shifts:write`; `schedules:write` alone only exposes schedule-level actions.

Schedule editing is board-triggered. Published rows expose a two-step Reopen action for authorized publishers; reopening returns the schedule to draft, hides it from staff, and re-enables corrections until republish. Publish results explicitly distinguish pending automatic notification delivery from terminal failure even though the schedule commit succeeded. Clicking an empty time slot opens the shift form prefilled for that person and time. Existing shifts expose edit and two-step delete actions. Shift creation requires staff, location, role, date, and times, and board drag edits persist through tenant-scoped shift endpoints. Shift and demand-window end times earlier than their start time serialize on the next local calendar day; equal times remain invalid. Ambiguous wall times during daylight-saving fallback are rejected with validation instead of silently selecting a fold. Persisted overnight records hydrate back to the start date with their next-day end wall-clock time.

`?focus=open` filters the shift table and schedule board to unassigned shifts, matching the dashboard "Assign open shifts" links. `?date=YYYY-MM-DD` seeds the initial calendar date when provided. Without that parameter, the route renders a deterministic date placeholder and waits for browser hydration before selecting the browser-local day or issuing the board request, preventing server/client timezone drift. `?location=<uuid>` selects that location only when it appears in the tenant-scoped board response; malformed, missing, or inaccessible IDs fall back to the first visible location before scheduling data is projected.

Manual shift creation passes the tenant-visible, location-matched `DRAFT` schedule whose window contains the complete shift interval. When no containing draft is loaded, the API fallback may create one spanning every touched local calendar day; overnight fallback state mirrors that complete window instead of synthesizing a one-day schedule.

The v2 board returns at most 500 deterministic location options and reports truncation explicitly. The calendar does not auto-drain continuation pages or issue hidden location fan-out.

Schedule publication first reads the API's authoritative aggregate preflight and presents its schedule version, schedule cost, matching-webhook count/unit/aggregate cost, wallet balance, and projected balance before confirmation. The confirm click refreshes readiness and preflight; any version or configured-cost change requires confirmation again, and insufficient aggregate credits never send the publish POST. The exact accepted contract is sent in the publish body and bound to the `Idempotency-Key` payload fingerprint. Publishing requires an active paid subscription plus enough separately purchased or administrator-granted credits. Each schedule keeps one contract-bound key across authentication refresh, timeout, lost response, and deliberate replay; a changed reconfirmed contract rotates the attempt. Known payment or state rejection returns to preflight, while an ambiguous transport/server outcome exposes a direct same-key replay so an already committed settlement cannot be debited twice. A successful response must include a matching published schedule and internally consistent atomic settlement; the UI reports the exact credits consumed and resulting balance alongside any notification-delivery warning.

Ownership follow-up: `page.tsx` remains a large workflow aggregator spanning loading, editing, solving, break generation, publication, and recovery. New transport identity stays in pure helpers; a later refactor should split page-owned hooks by mutation workflow without duplicating scope guards or recovery state.
