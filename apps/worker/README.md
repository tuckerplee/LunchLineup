# Worker Service

Background job consumer for scheduling, PDF, billing, email, and webhook jobs.

## Files

- `README.md`: this folder guide.
- `main.py`: RabbitMQ consumer with production fail-closed dependency startup, broker-owned ready/retry/DLQ telemetry and terminal-transition metrics, bounded sanitized failure classifications and allowlisted `WORKER_FAILURE_*` persisted status codes, queue validation, retry handling, authoritative malformed-schedule terminal settlement, supervised durable background sweeps with cancellation-safe shutdown, Tenant-then-scheduling-advisory solve-result barriers, centralized exact metadata/debit/refund terminal replay validation, terminal refunds derived from immutable ledger rows, tenant-visible solve-job state updates, gRPC engine client, deterministic solved-schedule persistence with one atomic draft revision increment, and worker metrics server.
- `requirements.txt`: Python runtime dependencies for the worker.

## Folders

- `src/`: worker support modules.
- `tests/`: worker unit tests.

## Operations

The Compose healthcheck probes the worker Prometheus endpoint on `WORKER_METRICS_PORT`, which defaults to `3003`, and requires `lunchlineup_pdf_parser_ready` to be `1`. The supervised worker probes the isolated parser every bounded `WORKER_PDF_PARSER_HEALTH_POLL_SECONDS`; parser loss after startup therefore makes the worker unhealthy and emits `lunchlineup_pdf_parser_health_probe_failures_total` for alerting. Enabled durable sweep tasks are supervised alongside RabbitMQ consumption; if one exits unexpectedly, the worker cancels its peers, closes RabbitMQ, and exits so the metrics healthcheck cannot remain green.

Availability PDF bytes cross a bounded Unix socket into the dedicated `pdf-parser` container. That service has no network, production secrets, upload-volume mount, or writable root; it receives one document in private tmpfs, runs `pypdf` in a resource-limited child, returns a bounded result, and exits so Compose destroys any descendant process before another tenant document is accepted. Durable availability claim, retry, terminalization, refund, and cleanup ownership remains in the worker. Schedule-solve claim and result commit both require `ACTIVE`, a non-`FREE` plan, a nonblank Stripe subscription ID, and `stripeSubscriptionCurrentPeriodEnd` strictly after the database clock; credits never replace stale or missing paid-through state. Claim and commit independently require one exact positive reserved debit whose immutable `balanceAfter` equals `creditConsumption.newBalance`. Exact-once refunds update the wallet and persist their deterministic ledger row with the same atomic post-refund `balanceAfter`. Malformed schedule envelopes with a valid durable job ID resolve tenant, schedule, and location from the authoritative row under platform-admin capability, then lock the tenant before attempting the same exact terminal settlement. Tokenless retry handoff and terminal recovery may mutate only an unowned job or one whose execution lease has expired, preventing a failed or rejected stale delivery from clearing or refunding work held by a live replacement.
Production startup requires explicit `RABBITMQ_URL`, `ENGINE_GRPC_URL`, and `DATABASE_URL`; the worker refuses guest RabbitMQ credentials in production instead of falling back to localhost development defaults. Retryable failures are routed through durable TTL retry queues (`WORKER_RETRY_QUEUE_PREFIX`) before dead-lettering. Every bounded `WORKER_QUEUE_DEPTH_POLL_SECONDS` interval, the worker passively samples the ready queue, every declared retry queue, and the durable DLQ into `lunchlineup_solver_queue_messages{state="ready|retry|dead_letter"}`; `lunchlineup_solver_queue_depth` remains the ready-plus-retry compatibility total. `lunchlineup_solver_queue_telemetry_available` is `1` only after a complete broker snapshot and becomes `0` on refresh failure, so consumers report unknown instead of a stale or false zero. Each confirmed terminal reject increments `lunchlineup_solver_terminal_transitions_total`, allowing alerting both while one poison item remains in the DLQ and immediately after a terminal transition. Schedule solve deliveries use a durable single-owner token and bounded `WORKER_SCHEDULE_SOLVE_EXECUTION_LEASE_SECONDS` lease; concurrent duplicates are delayed without consuming retry budget, expired owners can be replaced, and stale owners cannot persist results or terminalize/refund the replacement. Auto-schedule jobs update `ScheduleSolveJob` rows through `RUNNING`, `RETRYING`, `SUCCEEDED`, `FAILED`, and `DEAD_LETTERED`; engine infeasibility is terminal instead of an infrastructure retry, while retry and terminal `statusReason` values are stable allowlisted `WORKER_FAILURE_*` codes and never raw exception text. Terminal state changes and exactly-once wallet refunds share one PostgreSQL CTE, and a terminal status write must succeed before the source message is dead-lettered. Solve claims lock `Tenant` before the job; solved-result persistence locks `Tenant`, the tenant scheduling advisory key, the draft schedule, then the job and shifts, matching API scheduling mutations. Before advancing a non-terminal job to `RUNNING`, claim requires the full current paid-subscription tuple and the centralized provenance validator requires an exact three-field credit record with positive bounded `consumedCredits`, nonnegative bounded `newBalance`, one matching tenant/amount/job-reason/post-debit-balance ledger row, and no deterministic refund. Invalid provenance stays nonterminal and is retried rather than being directly dead-lettered. A terminal replay still validates locked job, tenant, schedule, and location identity plus complete debit/refund provenance, but it does not reapply current subscription entitlement and performs no write. Result persistence rechecks both paid entitlement and debit provenance before reading shifts or incrementing the revision. Terminal replay applies the same provenance proof: `SUCCEEDED` must have no refund, while `FAILED` and `DEAD_LETTERED` require the exact deterministic refund and immutable post-refund balance. Missing, malformed, cross-tenant, mismatched, duplicate, wrong-reason, wrong-balance, or status-conflicting reservations fail before terminal state; concurrent retries accept only the exact existing settlement. Solved shifts, breaks, one guarded draft revision increment, result count, and `SUCCEEDED` state commit in one PostgreSQL transaction. Terminal replay exits before domain or revision writes; integer overflow rolls back the result transaction. A crash after commit is idempotent because redelivery observes the terminal durable job only after validating its provenance.
Schedule solve payloads carry exact `demand_windows`, `staff_skills`, legacy `skill_requirements`/`daily_demand`, named-weekday availability, explicit per-staff `availability_configured`, tenant-wide `existing_weekly_minutes`, bounded tenant-wide `existing_shifts`, and an IANA `timezone`. Empty or unconfigured employee availability is preserved through protobuf as unavailable, never unrestricted. Existing minutes use Monday dates in the schedule location's calendar. Minutes and exact cross-location intervals travel through separate dedicated gRPC request fields so weekly limits and overlap exclusion remain independent. Before persistence, the worker coalesces touching same-user/same-role solver segments, rejects true overlaps, carries `BREAK1`/`LUNCH`/`BREAK2` identities into persistence, splits exact and daily demand at every shift and break boundary, and rejects results whose actually working or qualified staff fall below demand. Payloads also carry the queued monotonic schedule revision and active-shift revision snapshot. Persistence locks and compares both snapshots before replacing shifts, so later manual break or shift edits fail stale instead of being erased.

## Testing

Run the worker suite from the worker package root so its production-style `src` imports resolve:

```bash
cd apps/worker
python -m unittest discover -s tests
```

`billing.sync` queue messages and the periodic usage sweep share the durable `StripeUsageEvent` dispatcher in `src/billing_usage.py`. Active paid tenants receive a stable active-staff snapshot every `STRIPE_USAGE_SNAPSHOT_INTERVAL_SECONDS` (five minutes by default), bounding usage freshness at subscription renewal. Snapshot preparation upserts by the immutable tenant, metric, period-start, and period-end tuple, so replay or reconciliation can rotate Stripe transport identities without creating another logical row. The sweep selects due retries before new snapshots, excludes tenants that already have a snapshot in the current interval, and orders new work by the least-recently metered tenant plus tenant ID so batches progress deterministically without repeatedly selecting the same leading tenants. Due rows are claimed with `FOR UPDATE SKIP LOCKED`, stale `SENDING` leases below the attempt cap are recoverable, and an expired final-attempt lease is atomically dead-lettered with an unknown-outcome marker so it cannot become stranded. Ordinary HTTP retries preserve transport identity for unknown outcomes. API-reconciled asynchronous rejections arrive already rotated. Retryable failures move to `FAILED`, and terminal failures move to `DEAD_LETTERED`. Production metering requires `STRIPE_METERED_USAGE_ENABLED=true`, `STRIPE_SECRET_KEY`, `STRIPE_METER_EVENT_NAME`, `STRIPE_METER_AGGREGATION=last`, and `DATABASE_URL`.

Dead-letter replay is disabled by default. An operator can set `STRIPE_USAGE_DEAD_LETTER_REPLAY_ENABLED=true` to let the periodic sweep atomically requeue an aged batch. `STRIPE_USAGE_DEAD_LETTER_REPLAY_BATCH_SIZE` defaults to 25, `STRIPE_USAGE_DEAD_LETTER_REPLAY_MIN_AGE_SECONDS` defaults to 900, and `STRIPE_USAGE_DEAD_LETTER_MAX_REPLAYS` defaults to one replay round. Selection uses `FOR UPDATE SKIP LOCKED`, requires last-value aggregation, rotates both Stripe transport identities, and records immutable logical identity, replay count/time, prior identities, and prior error in event metadata. The original timestamp and value remain unchanged, so last-value replay cannot add usage. The worker logs the requeued count. Disable the flag after the recovery window; events that exhaust the replay-round cap remain dead-lettered for investigation.

Password-reset email delivery uses the encrypted `PasswordResetEmailOutbox` dispatcher in `src/password_reset_email.py`. The sweep claims rows with `FOR UPDATE SKIP LOCKED`, recovers expired leases, and rejects suspended/deleted users, non-active tenants, and provider-suppressed recipients both at claim time and under the final row lock immediately before provider handoff. Blocked rows erase their encrypted payloads. Lease-owned transitions use the outbox ID as the Resend idempotency key and retry transient delivery failures up to `PASSWORD_RESET_EMAIL_MAX_ATTEMPTS`. Database and whole-cycle failures leave the loop alive for the next bounded interval instead of terminating the task. `lunchlineup_password_reset_email_sweep_running`, `lunchlineup_password_reset_email_sweep_ready`, `lunchlineup_password_reset_email_sweep_last_success_unixtime`, and `lunchlineup_password_reset_email_sweep_failures_total` expose task liveness and sweep readiness. Terminal rows increment `lunchlineup_password_reset_email_total{status="dead_lettered"}`, remain visible in `lunchlineup_password_reset_email_dead_lettered`, and emit token-free error logs for alerting. Bounded provider rejection accounting sets `lunchlineup_password_reset_email_systemic_provider_failure` only after the configured recent-failure threshold, so one bad recipient dead-letters without declaring a provider outage while a systemic outage fails worker readiness. Opaque `email.send` jobs with a validated durable `outbox_id` use the same dispatcher. Unsupported `webhook.deliver` jobs fail closed because tenant webhook replay is owned by the API replay worker.

Staff invitation delivery is disabled by default for development. Production API and worker startup require canonical `STAFF_INVITATION_OUTBOX_ENABLED=true`; email-addressed user creation fails atomically instead of committing without a durable delivery intent. Enabled worker startup also fails closed unless `DATABASE_URL`, `PLATFORM_ADMIN_DB_CONTEXT_SECRET`, `STAFF_INVITATION_OUTBOX_ENCRYPTION_KEY`, `RESEND_API_KEY`, `EMAIL_FROM`, and the Compose-propagated `APP_ORIGIN` are present and the dedicated invitation key decodes to exactly 32 bytes without reusing the password-reset key under hex, base64, or base64url encoding. `APP_ORIGIN` must be an origin only, without credentials, path, query, or fragment; production requires HTTPS. Invitation HTML and plain text link to its escaped `/auth/login` path without recipient identity, tokens, or other PII in the URL. The platform capability is established and verified on every database transaction before any cross-tenant claim or diagnostic query.

The sweep atomically claims at most `STAFF_INVITATION_SWEEP_BATCH_SIZE=25` due or expired-lease rows with `FOR UPDATE SKIP LOCKED`, uses `STAFF_INVITATION_LEASE_SECONDS=120`, and shares the schema-bounded `STAFF_INVITATION_MAX_ATTEMPTS=8` contract with the API. Lowering that shared limit terminalizes rows already at the new bound in locked batches of 100 instead of stranding them. The sweep runs every `STAFF_INVITATION_SWEEP_INTERVAL_SECONDS=10`; Resend calls use `STAFF_INVITATION_PROVIDER_TIMEOUT_SECONDS=10` and the deterministic `staff-invitation/<outbox-id>` idempotency key. Backoff starts at `STAFF_INVITATION_RETRY_BASE_SECONDS=15`, caps at `STAFF_INVITATION_RETRY_MAX_SECONDS=3600`, and applies `STAFF_INVITATION_RETRY_JITTER_RATIO=0.25`. Startup validates all numeric bounds and rejects unsafe relationships such as provider timeouts that can exhaust a lease, retry maxima below the base, health windows shorter than two sweeps, or stale bounds shorter than two sweep intervals.

Health accounting uses `STAFF_INVITATION_PROVIDER_HEALTH_WINDOW_SECONDS=300`, `STAFF_INVITATION_PROVIDER_FAILURE_THRESHOLD=3`, bounded counts capped by `STAFF_INVITATION_DIAGNOSTICS_COUNT_CAP=1000`, and `STAFF_INVITATION_SWEEP_MAX_STALENESS_SECONDS=60`. The worker healthcheck requires running and ready gauges, a fresh successful-sweep timestamp, and no systemic provider failure. Prometheus alerts separately identify terminal rows, provider outage, failed cycles, and stale cycles; Grafana shows readiness, freshness/configured bound, terminal count, provider state, and due/expired/recent-failure backlog. User or tenant deletion/suspension cancels before provider handoff, recipient suppression and permanent provider rejection dead-letter immediately, transient failures retain the encrypted payload for retry, and all terminal transitions rely on the database trigger to erase the envelope. Invitation and account-recovery delivery are operational access flows and never consume scheduling credits. Metrics use the `lunchlineup_staff_invitation_*` prefix and logs contain only fixed error codes, attempt counts, and opaque outbox IDs.
