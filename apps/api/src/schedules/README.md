## Notes

Staff schedule reads recognize both legacy enum claims and current RBAC role labels, then self-scope to published schedules containing an active assigned shift. Demand-window inputs and auto-schedule job snapshots require `schedules:write`; `schedules:read` never exposes team demand, staff snapshots, solver constraints, or credit-consumption details. `GET /v1/schedules/:id/publish/preflight` returns the server-derived schedule revision and cost, matching enabled webhook count and unit/aggregate cost, total configured cost, current balance, sufficiency, and the exact contract the manager must accept. Every scheduled shift create, update, delete, changed bulk/reassignment, setup update, generated break persistence, and manual break replacement increments the owning draft revision inside the shared Tenant-then-advisory mutation transaction; no-op and replay paths do not. Publishing hashes the accepted contract with the tenant and schedule, rechecks it under the scheduling lock, and returns 409 on revision or cost drift before any debit, audit, webhook delivery, notification outbox, or status write. A matching publish atomically commits separately identifiable schedule and webhook ledger debits plus durable delivery/notification intents, and stores the complete settlement in the publish audit replay record. A corrected reopen and republish creates a new staff notice, then attempts notification delivery only after commit; retryable rows are reported as pending while only dead-lettered rows count as failed. Publish and new auto-schedule transactions lock the same draft row, and publish locks/rejects active solve jobs before reading shifts, so a queued or running generation cannot race an older draft into publication. Long default-policy shifts require `BREAK1`, `LUNCH`, and `BREAK2`; medium shifts require `LUNCH`. Auto-schedule messages snapshot the monotonic schedule revision and active shift IDs/revisions so workers reject stale solver results, including results queued before a manual break edit.

Assigned staff must have at least one global or matching-location availability rule, and every shift segment must fit the contiguous union of those rules; touching and overlapping windows combine, while any real minute gap remains unavailable. Publish aggregates the candidate schedule with every other active tenant shift for each assigned employee using Monday-to-Sunday boundaries in the schedule location timezone. The most recent successful solve's `max_hours_per_week` applies when valid; otherwise the existing 40-hour solver default applies. Schedule creation, deletion, reopening, and demand-window replacement acquire the active-paid entitlement lock but perform no credit settlement, so they remain available at zero wallet balance. Publication preflight is also entitlement-only, while publication itself atomically verifies and charges the exact accepted aggregate schedule-plus-webhook cost. API shift mutations and publish take the same tenant scheduling transaction lock before the final locked read/recheck, so concurrent entitlement loss, last-credit consumption, and API edits cannot bypass the gate. This behavior uses existing rows and requires no persistence migration.

Only one nonterminal solve job is allowed per schedule transaction. Active recovery acquires `Tenant`, then the draft `Schedule`, then every active job in deterministic order; multiple candidates fail closed. Every nonterminal reuse revalidates current scheduling entitlement, which requires `ACTIVE`, a non-`FREE` plan, a nonblank Stripe subscription ID, and a paid-through timestamp strictly in the future. It then proves exact bounded positive credit metadata and one `schedule-credit-<jobId>` row whose tenant, negative amount, job-bound reason, and immutable `balanceAfter` match `creditConsumption.newBalance`, with no deterministic refund while active. A zero current wallet remains valid because that job's positive credit was already reserved exactly once; FREE, null/past paid-through, and other entitlement loss fail before solve-input reads or new writes. New solves perform the billable entitlement check before input snapshots, atomically decrement the wallet, persist the debit's post-decrement balance, and record the same value in job metadata. Terminal replay alone bypasses current entitlement, after proving successful jobs have no refund and failed/dead-lettered jobs have the exact deterministic refund with the expected post-refund balance; it performs no write. Missing, malformed, mismatched, duplicate, wrong-reason, wrong-balance, or debit/refund coexistence fails closed.

Schedule lists default to 100 rows, accept 1 through 200, and return pagination metadata with an opaque continuation cursor. Optional UTC `startDate`/`endDate` overlap and `locationId` filters preserve calendar windows; stable newest-first `(startDate, id)` ordering and the tenant/deletion/start-date composite index keep historical schedules out of first-load scans.

Schedule creation locks and revalidates the active location row before interpreting local dates or inserting a draft. Active schedule reads exclude schedules whose location was soft-deleted. Those schedules and their assignments remain retained in the database for audit and controlled export; the operational API does not present them as current work.

Solve-job API responses map any persisted status or publication failure text, including legacy rows, to fixed user-safe messages. Outbox publication remains durable through transient RabbitMQ failures. A failed broker publication becomes terminally eligible after `SCHEDULE_OUTBOX_MAX_PUBLISH_ATTEMPTS` (default `8`) or `SCHEDULE_OUTBOX_MAX_PUBLICATION_AGE_MS` (default 24 hours), whichever comes first, but tokenless publication recovery preserves `RUNNING` and issues no refund while a worker execution token has a live lease. After the lease expires, the recovery transaction locks the tenant then job, repeats the no-live-lease predicate in its terminal compare-and-set, proves exactly one deterministic debit with the same tenant, job reason, configured amount, and post-debit balance, derives the refund from that debit row, and increments the wallet only when the deterministic refund insert succeeds. Its immutable refund `balanceAfter` is the actual post-refund wallet at settlement time and may differ from the original debit baseline after same-tenant activity. Missing, mismatched, duplicate, or conflicting pre-publication provenance records a bounded publication failure while leaving job status and execution ownership unchanged; no terminal state is accepted without the exact debit plus status-appropriate refund proof. Concurrent terminal retries accept only an already-terminal job with an exact nonnegative stored refund settlement.
