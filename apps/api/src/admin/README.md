## Notes

Cancellation recovery terminalizes or excludes stale lifecycle barriers and changed-subscription intents before applying `LIMIT`, so they neither starve newer work nor inflate the actionable backlog. Each batch row is claimed immediately before provider work and renewed while the provider call is in flight; Stripe cancellation and compensation updates use distinct stable keys derived from the durable operation ID. An expired `PENDING_PROVIDER` platform intent remains claimable when a legal hold arrived during provider-result uncertainty, but recovery is provider-readback-only: an unscheduled subscription is never newly canceled, while the same operation marker on an already-scheduled subscription restores monotonic ownership and enters exactly-once compensation. Compensation eligibility is the monotonic durable ownership bit, established by this operation returning `scheduled` or by Stripe returning the same stable operation marker after an uncertain post-provider write. Attempt counts and an unmarked `already_scheduled` result never establish ownership, while a marked replay can never downgrade ownership already persisted as true. A later explicit customer cancellation is authoritative: it idempotently adopts the provider marker under its durable operation, and stale platform compensation recognizes only an exact-subscription customer intent in the real durable authority states `PENDING_PROVIDER`, `PROVIDER_APPLIED`, or `FINALIZED` before and after its provider write. Thus a customer request paused after Stripe success but before `markProviderApplied` already fences compensation; blocked, superseded, or mismatched operations do not. If takeover wins after the stale write starts, compensation reasserts the customer cancellation and marker under a separate deterministic repair key; response-loss retry remains exact and the platform operation finishes blocked as `not_owned`. A terminal subscription carrying another operation's marker is checked before `already_terminal`, and local compensation finalization revalidates the exact customer-finalized tenant/intent winner under the lifecycle lock, so it cannot downgrade `CANCELLED`, clear a replacement binding, revoke sessions, or alter credits. If a retention legal hold commits after a platform-archive mutation still owned by that operation, the intent changes durably to compensation-pending, cancellation is unscheduled outside the transaction only while no customer authority exists, and attributed blocking evidence is finalized under the canonical lifecycle lock. A terminal provider webhook protected solely by that winning hold fails closed to `PAST_DUE`, never local `CANCELLED`.

Platform-admin routes use `TenantPrismaService.withPlatformAdmin` for tenant-owned tables so forced RLS stays enabled for normal app traffic while cross-tenant operations remain explicit and transaction-local. Cross-tenant audit rows retain target-tenant `tenantId`/`userId` semantics while immutable non-relational actor IDs and request IP/user-agent identify the platform administrator.

User suspension writes `suspendedAt`, never `deletedAt`, and is reversible only for non-deleted users. Suspend and activate reject self-mutation before opening a transaction. Each mutation locks tenants before actor/target users, locks the exact request session, and revalidates live `admin_portal:access` in the same Serializable transaction. A dual-source system-admin target additionally requires a dual-source system-admin actor. Authorization denial precedes target, session, and audit writes, and any later denial rolls the transaction back. Suspension revokes every active target session and records attributed audit evidence atomically; activation rechecks paid-plan user capacity before clearing `suspendedAt`. Deleted-user tombstones remain irreversible, while an active platform administrator may recover a different suspended user.

Lost-factor recovery rejects self-reset, requires exact `reset-mfa:<user-id>` confirmation and a trimmed 10-to-500-character support reason, and locks the active target row before mutation. MFA factor removal, TOTP replay-claim deletion, revocation of every unrevoked target session, and attributed audit creation share one platform-admin transaction, so an audit failure rolls back the recovery.

Platform tenant creation requires an owner name and email. FREE workspaces default to `ACTIVE` with free-tier entitlements. Any `TRIAL` receives a concrete future `trialEndsAt`, bounded by `ADMIN_TENANT_TRIAL_DAYS` (14 by default, maximum 90), and every new tenant starts with zero usage credits. Paid plans cannot be created `ACTIVE` because this endpoint has no verified Stripe subscription or tenant-level manual entitlement proof. Tenant creation, owner creation, default RBAC role provisioning, Admin assignment, and audit logging share one platform-admin transaction so any failure rolls back the entire workspace.

Platform tenant archive first commits an attributed, idempotent lifecycle intent, then schedules any stored Stripe subscription for period-end cancellation outside a database transaction, and finally disables access plus records the final audit outcome in one transaction. Archive/cancellation and deletion use the same tenant lifecycle lock and recheck the durable deletion barrier before finalization, so an overlapping archive cannot replace `SUSPENDED` with `CANCELLED` or strand billing reconciliation; archive recovery also stops while a retention legal hold is active. Every API replica runs the same bounded reconciler, but the canonical PostgreSQL advisory lock plus per-intent leases admit only one owner. A durable provider result is reread before local finalization, while an uncertain post-Stripe write remains lease-bounded and safely replays Stripe's idempotent period-end update after expiry. Restore and platform activation share the same paid-plan eligibility check: the stored Stripe subscription must be active or trialing, owned by the tenant, and not scheduled for period-end cancellation. Their final mutation transaction takes that same canonical lifecycle advisory lock, then the tenant row lock, and rechecks both billing eligibility and the durable deletion barrier so committed phase-one `SUSPENDED` deletion cannot be overwritten. Tenants without Stripe can be restored or activated only when explicitly assigned the `FREE` plan.

Generic platform tenant edit accepts profile and bounded date fields only. It rejects `planTier`, `status`, `usageCredits`, `stripeSubscriptionId`, and `stripeSubscriptionCurrentPeriodEnd` before database access so billing entitlements cannot bypass Stripe coordination, wallet changes cannot bypass the idempotent credit ledger, and lifecycle state cannot bypass the dedicated suspend, activate, archive, restore, and delete routes. Plan create and update also reject included-credit fields or metadata; legacy `creditQuotaLimit` storage remains fixed to `NULL`.

Tenant-admin account lifecycle routes override the platform-admin permission with `account:data_export` for full export, `settings:write` for status, and `tenant_account:lifecycle` for cancellation/deletion, scoped only to the caller tenant through `TenantPrismaService.withTenant`. The protected export and lifecycle permissions are granted to the default system Admin role but remain reserved from tenant-created role grants.

Customer-initiated cancellation first commits an attributed internal intent under the canonical tenant lifecycle lock, schedules the stored Stripe subscription for `cancel_at_period_end` outside a database transaction, persists only a sanitized provider outcome, rereads/reasserts that state, and then records the final audit while keeping local access active through the paid period. If reconciliation instead observes authoritative `already_canceled`, the same finalization transaction projects `CANCELLED`, clears the subscription binding, removes paid feature entitlement, and records completed-cancellation audit evidence before returning. If a terminal webhook wins after provider return but before provider-result persistence, `markProviderApplied` recognizes the exact canonical local terminal state and atomically preserves the original intent, sanitized outcome, and one completed audit without repeating finalization or session/billing side effects. A finalized scheduled intent whose effective period has arrived becomes an actionable readback-only reconciliation row; missed-webhook recovery projects terminal provider truth while preserving the original operation ID, subscription fingerprint, one scheduled audit, and one completed audit. If the deleted webhook arrives first, API replay converges that same finalized intent locally without a second provider call or replacement intent. A finalized replay returns the durable provider result even after that canonical binding clear, without repeating provider work or audit writes. Every terminal deleted webhook revalidates the exact customer and current subscription binding under the billing transaction lock even when pre-transaction resolution found subscription A before a finalized marker existed; a concurrently bound subscription B is retained and A is audited as skipped. A late terminal webhook with a null binding can resolve only through the exact finalized customer intent, customer ID, tenant metadata, and provider subscription ID, while paused events and any replacement-subscription binding are audited without entitlement side effects. Normal webhook event ordering and unique-event idempotency still apply. Provider and finalization failures return only a pending-reconciliation classification; the autonomous reconciler discovers the durable backlog every 30 seconds by default, claims at most 20 intents, recovers expired two-minute leases, and retries without exposing provider IDs, statuses, or errors. `TENANT_CANCELLATION_RECONCILE_INTERVAL_MS`, `TENANT_CANCELLATION_RECONCILE_BATCH_SIZE`, and `TENANT_CANCELLATION_RECONCILE_LEASE_MS` are hard-bounded in process. Recovery is a free control operation and never enters usage metering or the credit ledger. Prometheus exposes bounded success/failure counters and the actionable backlog gauge, including due finalized schedules awaiting provider readback. Deletion is a separate durable two-phase saga. Phase 1 acquires the canonical tenant lifecycle lock before the tenant billing/row locks, writes the fail-closed `SUSPENDED` barrier with immutable actor IDs, revokes sessions, deactivates outbound webhooks, dead-letters queued or running schedule solves, refunds their wallet debits exactly once, and commits a due reconciliation row plus one exact replica owner/lease/fencing token. The scheduler claims one barrier just in time and excludes attempted tenants for the rest of that sweep; the deletion owner renews and revalidates the exact unexpired owner/token immediately before provider entry. A heartbeat retains ownership through deadline, claim-loss, or shutdown abort until the active provider transport actually terminates; only then may the exact owner/token be released with persistent exponential backoff. Provider, deadline, shutdown, and phase-2 failures clear only the matching owner/token, while `nextAttemptAt,barrierCreatedAt,tenantId` ordering and skip-locked claims let newer healthy barriers advance. Phase 2 rechecks the exact claim and barrier, then records `PURGED`, `deletedAt`, retained-record timing, billing cleanup, and the same actor IDs in one transaction. Thirty-day minimization clears payload/IP/user-agent and direct `userId` while retaining tenant attribution and a deterministic pseudonymous `actorUserId`. A post-barrier failure returns a `PENDING_BILLING_CLEANUP` receipt with the original request timestamp, never a false completion or provider error; a finalized retry returns the existing `FINALIZED` receipt without repeating external work. Every API replica runs the independent deletion-billing reconciler immediately at startup and then every 30 seconds by default, attempting at most 20 due barriers one at a time. `TENANT_DELETION_BILLING_RECONCILE_INTERVAL_MS`, `TENANT_DELETION_BILLING_RECONCILE_BATCH_SIZE`, `TENANT_DELETION_BILLING_RECONCILE_LEASE_MS`, `TENANT_DELETION_BILLING_RECONCILE_ATTEMPT_TIMEOUT_MS`, and `TENANT_DELETION_BILLING_RECONCILE_STOP_DRAIN_MS` are hard-bounded. Prometheus exposes deletion-specific bounded outcomes, backlog, oldest-pending age, last sweep, last healthy sweep, and maximum staleness without sharing cancellation metric names. The first `application_data` retention page remains an operator-visible discovery/retry path, but it is not the only scheduler owner. Verified webhooks cannot restore entitlement while the barrier is active. Billing support handles any off-cycle refund decisions.

Tenant deletion is two-stage. The scheduler service token may execute only the separately confirmed `application_data` stage after 30 days using API server time; that daily run also deletes one bounded batch of sessions expired beyond a 24-hour grace or revoked for 30 days, with refresh replay rows cascading and active credentials excluded. Time cards, their breaks, and approval rows are removed only through the capability-gated payroll operational purge inside the same platform-admin transaction. That function rejects open cards, any non-`LOCKED` payroll period, or a card without a current immutable locked snapshot, so rejection rolls back all cleanup and leaves `applicationDataPurgedAt` unset. Billing, Stripe usage, credit, immutable payroll evidence, audit, backup, and security records and the tenant tombstone remain, but retained provider metadata/errors and direct audit identities/payloads are minimized. Audit action/resource/timestamp and tenant attribution remain compliance evidence with pseudonymous actor attribution. The same identity may dry-run `retained_records` discovery after seven years but cannot delete it. Only a platform admin can execute that distinct stage or hard-delete route after external retention checks. Retained payroll and audit rows are removed through separate security-definer database functions that independently verify platform-admin capability and tenant eligibility; payroll rows are deleted in FK-safe order before user and tenant cleanup.

Retention candidate reads use stable `deletedAt,id` keyset pages of at most 25 candidates. The first application-data page reports the global dormant-session eligible/purged counts; each tenant then gets a separate 60-second platform-admin transaction guarded by a PostgreSQL advisory transaction lock and an eligibility re-check. Responses expose processed, skipped, and failed counts plus `nextContinuation`; the scheduler wrapper scans bounded continuation pages and fails the overall run after isolated attempts if any tenant failed. Purge and billing-reconciliation attempts return fixed failure classifications rather than raw database, provider, URL, credential, or filesystem exception text.

Platform administrators may place or release a tenant retention legal hold even after deletion is requested. Both transitions require a 10-to-500-character reason, acquire the canonical tenant lifecycle advisory transaction lock shared by purge, cancellation intent, and direct user deletion triggers, CAS the current hold state, and write attributed audit evidence in the same bounded transaction. Candidate queries exclude active holds; direct and scheduled purge paths recheck after taking the lock. A hold that wins the lock makes a concurrent restricted-role user tombstone fail PostgreSQL `42501`, rolling back user scrubbing, auth invalidation, and audit redaction together.

Tenant exports are durable asynchronous NDJSON jobs. PostgreSQL persists requesting tenant/user authorization, queue state, writer lease, expiry, progress/error, opaque artifact key, quota bytes, and explicit cleanup state/owner/lease/attempts. A global advisory lock atomically reserves each writer's full ceiling; a job advisory lock fences file creation against expiry cleanup. Expiry or generation failure first commits `PENDING` cleanup while retaining artifact identity, writer claim, and quota. The cleanup owner then unlinks every deterministic final/partial path, verifies absence, syncs the artifact directory on supported hosts, and only afterward commits `COMPLETE` with zero bytes and cleared ownership. Deletion failure returns the claim to retryable pending state without releasing quota; database triggers reject premature ownership mutation and block tenant-export row purge while plaintext or writer ownership remains. Global and per-tenant admission counts every non-complete positive reservation even after expiry, preventing new over-quota claims. Public status reports bytes only for ready artifacts. `TENANT_EXPORT_MAX_ARTIFACT_BYTES` defaults to 256 MiB, `TENANT_EXPORT_PER_TENANT_QUOTA_BYTES` to 512 MiB, and `TENANT_EXPORT_GLOBAL_QUOTA_BYTES` to 2 GiB. `GET /admin/account/exports` returns at most five unexpired requester-owned jobs. Production requires explicit absolute shared storage. Artifacts use `0600`, the directory uses `0700`, orphan cleanup protects all database-owned paths, internal lifecycle-intent settings are never exported, and downloads remain attributed, private, non-cacheable, and `nosniff`.

The same repeatable-read export covers time-card payroll-period/time-zone/revision state plus payroll policy versions, periods, approvals, locked entries, amendments and decisions, operation audit summaries, export batches and lines, reconciliation receipts, events, and current line states. Cursor pages contain only stable identity fields; each full projection is fetched and streamed separately, and JSON strings are emitted in bounded chunks so one valid record larger than 4 MiB does not fail the export. Every collection is tenant-filtered and stably ordered. Request hashes, operation identifiers, lock request hashes, and stored idempotency responses remain internal; customer-visible rows retain actors, decisions, timestamps, statuses, counts, external references, and evidence hashes needed to audit payroll operations.

Platform tenant and credit-list reads are bounded keyset pages ordered by createdAt and id, fetch limit plus one, and reject malformed limits, cursors, and overlong or control-character searches before querying. Tenant search exposes only existing name and slug fields. Credit balances and ledger history have independent cursors, and responses expose page metadata without claiming global totals.
