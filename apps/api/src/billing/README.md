## Credit Purchase Policy

Credit-pack API responses never expose Stripe Price IDs. Subscription plans and trials grant zero credits; a pack grant occurs only after its separate paid Checkout is authoritatively verified.

## Notes

One-time credit packs use distinct `STRIPE_PRICE_CREDIT_PACK_100`, `STRIPE_PRICE_CREDIT_PACK_500`, and `STRIPE_PRICE_CREDIT_PACK_2000` Price IDs. `GET /billing/credit-packs` returns only fixed pack codes, credit counts, configuration state, and Stripe-authoritative amount/currency for configured active one-time Prices. `POST /billing/credit-packs/checkout` and both synchronous and webhook fulfillment require canonical `ACTIVE`, non-`FREE`, nonblank subscription identity, and future `stripeSubscriptionCurrentPeriodEnd` under the tenant billing lock. Payment Checkout metadata binds the tenant, pack code, credits, Price ID, unit amount, currency, and quantity as an immutable catalog snapshot; safe server-controlled return URLs, bounded idempotency, and exact-session reuse prevent caller-controlled redirects and duplicate open purchases. Verified `checkout.session.completed` and delayed `checkout.session.async_payment_succeeded` handlers re-read the Session and line item, validate every monetary field against that snapshot, and verify customer, PaymentIntent, and current subscription. A completed but unpaid Session records only a recoverable observation; a later authoritative paid event remains eligible to verify and grant. A paid Session delivered after FREE downgrade or paid-through expiry records refund-required state and reaches a verified provider refund with zero wallet or ledger grant. Rotating or archiving the configured Price after Checkout creation does not strand an older paid Session. A successful eligible payment reserves `stripe-credit-purchase-<session id>`, settles the pack through the shared debt-first owner, records immutable wallet and debt results in `CreditTransaction`, and records the billing event in one tenant transaction; outstanding debt is repaid before any remainder enters `Tenant.usageCredits`. Replay validates the Session-bound tenant, pack amount, reason, and stored balance, then returns that original settlement even after intervening wallet activity; nullable or malformed legacy settlement rows fail closed.

Every paid credit purchase has one durable terminal outcome: the deterministic credit ledger grant or one deterministic `refund_confirmed` billing-event row after Stripe authoritatively reports a full-amount, exact-currency `succeeded` refund. Provider-refund fulfillment never mutates the usage-credit wallet and therefore emits no `CreditTransaction`; observing a purchase ledger row instead is a conflicting terminal outcome and fails closed. Any paid purchase that cannot be granted, including deleted, suspended, inactive, ownership-mismatched, or snapshot-mismatched fulfillment, first commits a durable `refund_required` purchase row before calling Stripe. Refund creation uses Checkout-Session-and-attempt-stable idempotency keys, records only allowlisted refund state, and re-fetches Stripe state after response loss. `refund.created`, `refund.updated`, and `refund.failed` webhooks enter the same reconciler; their authoritative refund ID takes precedence over a stale stored failed or canceled refund ID, is re-fetched and fully validated, and can terminalize a provider-created replacement without issuing another refund. `pending`, `requires_action`, `failed`, `canceled`, unverifiable, and provider-error outcomes stay recoverable and force webhook failure rather than returning a safe acknowledgement; failed/canceled refunds receive at most five deterministic automatic attempts and then remain operator-blocking until authoritative recovery. Duplicate, concurrent, and out-of-order Checkout or refund events cannot double-grant, double-refund, or create a second terminal outcome. Provider payloads and messages are never persisted or logged.

Production billing requires `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_WEBHOOK_ENDPOINT_ID`, `APP_ORIGIN`, configured metered Stripe price IDs such as `STRIPE_PRICE_STARTER`, `STRIPE_PRICE_GROWTH`, or `STRIPE_PRICE_ENTERPRISE`, and metered usage settings `STRIPE_METERED_USAGE_ENABLED=true`, `STRIPE_METER_EVENT_NAME`, and a snapshot interval no greater than five minutes. The production launch gate retrieves the live meter, prices, and webhook endpoint to verify payload mappings, meter attachment, live mode, active state, and all handled entitlement subscriptions. `STRIPE_BILLING_PORTAL_CONFIGURATION_ID` may select a pre-created portal configuration, but the API verifies that it disables subscription updates; without one, the API reuses or creates its own safe payment, invoice, and cancellation configuration. Tenant-facing routes expose the feature matrix, configured Stripe Checkout price options, a safe billing-portal redirect, capacity-checked plan changes, and paused-subscription resume. Checkout and server-controlled plan changes reject inactive target plan definitions under the tenant billing lock before Stripe mutation. Checkout preflights plan capacity, omits licensed quantity for metered prices, reuses only an open session for the exact selected price, expires incompatible tenant sessions, and uses a tenant-and-price-scoped bounded idempotency key. First Stripe customer creation uses the same lock plus a stable Stripe idempotency key and a conditional database claim.

Checkout completion, invoice payment, and invoice payment-failure handlers retrieve and verify the current Stripe subscription before changing ownership, status, or plan. If an invoice arrives before the subscription webhook has bound `stripeSubscriptionId`, the API resolves the unique tenant from the verified subscription metadata and Stripe customer, conditionally claims the unbound subscription under the billing locks, and throws before recording the event when ownership is ambiguous or conflicting so Stripe retries it. This lets `no_payment_required` and `unpaid` Checkout sessions bind their subscription safely, resolves proration invoices from the current subscription item instead of invoice-line order, and keeps the active old plan during a failed `pending_if_incomplete` update while still restricting a genuinely delinquent renewal. Transient Stripe lookup failures abort before the unique `BillingEvent` is stored so Stripe can retry the same event ID; deterministic stale or already-bound identity mismatches remain audited with a skipped disposition.

Append-only `BillingEvent` rows retain a durable per-subscription event high-water mark ordered by Stripe creation time and restrictive-event precedence. Canonical subscription events check that cursor while holding tenant and subscription advisory locks, then retrieve authoritative Stripe state under the same locks when newer or when a distinct event shares the same second; Stripe event IDs do not decide equal-second subscription state. Terminal state still wins equal-second conflicts, while authoritative paid or active state wins over recoverable `past_due`, `unpaid`, or `incomplete` payment failure in either same-second delivery order. Older-second events cannot reactivate a cancellation. Stale or unverified events remain audited but only successfully applied events advance the cursor. Created, updated, resumed, invoice-activated, and period-end-cancellation synchronization writes `Tenant.stripeSubscriptionCurrentPeriodEnd` from authoritative `subscription.current_period_end`; missing/invalid values fail closed to `NULL`. `incomplete_expired`, deleted, purged, or otherwise removed subscriptions clear both that period boundary and `stripeSubscriptionId`. Paused subscriptions retain their ID in authenticatable `PAST_DUE` state; the resume route either synchronizes an immediately active subscription or returns a hosted invoice or billing-portal payment URL. Billing feature responses expose `subscriptionRecoveryAction`, so only Stripe-confirmed `paused` subscriptions use resume while non-paused delinquency uses portal recovery. `invoice.finalization_failed` retrieves and synchronizes the current subscription under the same locks, preserving paid access while Stripe reports the subscription active and restricting access only when the authoritative subscription is delinquent. Portal plan switching is disabled; `/billing/change-plan` preflights capacity, and any out-of-band over-capacity Stripe downgrade converges to the lower local plan with free-tier entitlements instead of retrying the webhook forever. Plan and capacity presentation can recognize an unexpired trial, but Stripe `trialing` synchronizes to local `TRIAL` and is never paid entitlement for billable features, even when the tenant has a positive externally granted wallet balance. Delinquent, suspended, cancelled, and purged tenants cannot start billable feature work.

Billable value-producing feature access requires `Tenant.status = ACTIVE`, a nonblank `Tenant.stripeSubscriptionId`, `Tenant.stripeSubscriptionCurrentPeriodEnd` strictly in the future, `Tenant.creditDebt = 0`, and sufficient separately purchased usage credits. Missing, invalid, or stale authoritative paid-through state fails closed even when the wallet is positive; `gracePeriodEndsAt` is never entitlement proof. Stripe synchronization is the sole writer and must populate or clear the exact `stripeSubscriptionCurrentPeriodEnd` field with the subscription binding. Subscriptions do not provide recurring included credits or unlimited usage. Entitlement-only control, read, and recovery operations require the active paid feature but do not require a positive wallet or zero debt and never call the credit settlement path. The wallet debit and durable request reservation share one transaction for value-producing operations; zero credits or outstanding credit debt block a new operation, while completion or recovery of an already charged request does not consume a second credit. Schedule and lunch/break reservations preserve their historical deterministic transaction IDs but delegate the actual wallet mutation to the same metering primitive as generic feature usage. Every positive credit settlement first reduces debt and exposes only the remainder as spendable wallet value. Stripe purchases and administrative grants delegate to the shared TypeScript settlement owner; tenant-deletion, user-deletion, schedule, webhook, and availability-import refunds call `public.settle_positive_credit_value` inside their existing transaction so NestJS and Python cannot diverge on that rule. Administrative grants remain a separate platform-admin correction path at `POST /admin/credits/grant`, require an idempotency key, and are the only API path for non-purchase credits. New public or platform-provisioned tenants always start at zero. Generic tenant edits cannot mutate wallet balances, credit debt, Stripe subscription identity, or authoritative paid-through state, and plan definitions cannot carry included, unlimited, quota, or wallet-credit metadata.

Administrative grants require the exact live platform actor and session to pass canonical in-transaction RBAC authorization. One Serializable platform-admin transaction locks the target tenant, repays outstanding debt before incrementing the wallet, writes deterministic `CreditTransaction.balanceAfter` and `debtAfter` results, and writes an attributed `TENANT_CREDITS_GRANTED` audit. Audit failure rolls back wallet, debt, and ledger. Replay reauthorizes the current actor, validates the original total value across wallet and debt deltas plus the deterministic audit, returns the stored original wallet balance even after later activity, and emits neither another grant nor another audit. Feature-use replay follows the same immutable wallet-and-debt rule; malformed legacy settlement rows fail closed instead of returning current state.

Tenant deletion first commits `SUSPENDED` as a fail-closed barrier plus a durable deletion-billing reconciliation claim before entering this provider facade. A paired replica owner/lease and fencing token prevent request/scheduler or multi-replica concurrent entry; only the exact owner may finalize local settlement, while persistent attempts, exponential backoff, and fair due ordering recover failures without starving newer barriers. Billing cleanup then uses a short database transaction to acquire the billing advisory lock and snapshot only a `SUSPENDED` or already `PURGED` tenant before releasing the transaction and making Stripe calls. Every Checkout expiration, subscription cancellation, and Customer deletion carries a distinct deterministic idempotency key derived from the immutable deletion operation ID and provider object ID. Stripe's Node request options do not accept a caller `AbortSignal`, so every purge read and mutation instead receives a bounded SDK transport timeout capped by the absolute provider-attempt deadline and disables SDK network retries. Deadline, claim-loss, and shutdown abort the local operation, but the deletion heartbeat and shutdown drain retain the exact claim until the in-flight SDK request has terminated; transport timeout/readback convergence finishes before claim release, so another replica cannot overlap provider mutation. It expires every verified open subscription Checkout session and owned credit-pack payment Checkout session, then verifies and immediately cancels every owned nonterminal subscription. After all billable objects are terminal, cleanup verifies the Stripe Customer's tenant metadata, deletes the Customer to remove provider-side PII, confirms Stripe's deleted state, and only then clears both local Stripe bindings under the billing lock. Stripe or database failures retain a durable local binding and abort final deletion so the whole cleanup retries; a retry that observes the same Customer already deleted completes only the guarded local clear. Paid credit-pack completion that races or follows the barrier is refunded idempotently before acknowledgement and can never grant credits, including after the deleted Customer binding has been cleared. Subscription webhooks received while `SUSPENDED` are audited without entitlement side effects, and a later subscription webhook for a purged tenant can only resolve through authoritative customer plus tenant metadata; the handler cancels that verified subscription, retains the purge state, and records the disposition without restoring entitlement.

Metered usage persists tenant-scoped `StripeUsageEvent` rows before posting active-staff meter events to Stripe and records `submittedAt` on every send claim. Each tenant, metric, period-start, and period-end tuple is immutable and unique; Stripe identifiers and HTTP idempotency keys are transport identities that may rotate without allowing a second logical snapshot. `STRIPE_METER_AGGREGATION=last` is a required runtime contract matching the live meter launch gate. A separately signed thin-event destination delivers Stripe's asynchronous meter rejection events to `/billing/meter-errors/webhook`; the API fetches the authoritative v2 event and reconciles aggregate report samples by exact durable identifier or idempotency key. `v1.billing.meter.no_meter_found` is parsed separately from its top-level `reason.request` correlation and fails closed unless exactly one durable row is identified. Every asynchronous rejection, including a previously submitted row already marked `FAILED` after a local timeout, rotates both independently unique Stripe transport identities while retaining `logicalUsageIdentity`, the Stripe error event ID, and rejected transport identities in metadata. Duplicate webhook delivery is acknowledged only from that recorded Stripe event ID without a second transition.

When Stripe's declared aggregate `error_count` exceeds its capped samples, the API scans `SENT` or `SENDING` rows for the configured event name in the authoritative validation window. Explicit samples retain their code-specific bounded-retry or terminal disposition. If extra candidates make the remaining rejection identity ambiguous, the API requires last-value aggregation and moves every bounded candidate to retry with fresh transport identities, original timestamp and value, and an `ambiguous_bounded_retry` disposition. Replaying a valid candidate is non-additive under last-value aggregation. The webhook acknowledges once at least every declared rejection is durably covered; missing candidates, oversized ambiguity sets, non-last aggregation, and lost conditional updates fail closed. Conditional mutations retain tenant, observed status, prior identifier, prior idempotency key, and submission timestamp guards. The worker prepares periodic snapshots, claims due rows with a lease, and processes these durable states. Local development can boot without Stripe keys, but billing, webhook, cancellation, and metered usage calls return service-unavailable until configured.

Operational Stripe failures are logged as bounded JSON diagnostics containing only a fixed event, allowlisted error class and code, normalized category, and validated Stripe request reference. Exception messages, stacks, URLs, headers, payloads, credentials, and tenant or user identifiers are never included. Persisted metered-usage failure state uses the same bounded diagnostic and never stores raw provider exception text.

`stripe-credit-purchase.service.ts` remains an oversized billing orchestration hotspot because Checkout creation, fulfillment verification, and durable refund recovery currently share one provider client and tenant-lock boundary. A later bounded refactor should extract the append-only refund journal/reconciler behind a narrow interface; this P1 fix keeps that contract local to avoid changing schema or cross-module ownership while funds are at risk.
