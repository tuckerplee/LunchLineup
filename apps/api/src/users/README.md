## Security Notes

Role delegation is subset-based: a caller may invite or assign only roles whose effective permissions are all present in the caller's current server-resolved permission set. User role replacement and custom-role create/update/delete lock `Tenant` before users, the exact request session, assignments, roles, and permission edges in one tenant-scoped Serializable transaction; stale sessions, self changes, equal/higher-target changes, unheld or protected grants, foreign-tenant role IDs, and assignment overflow fail closed. Every replacement reconciles `User.role` to the highest active assigned system legacy role, falling back to `STAFF`, and any effective access change revokes all active target sessions before audit and commit. Mutating an existing dual-source system administrator or delegating system-admin authority requires both a current actor `User.role` of `SUPER_ADMIN` and an active system `SUPER_ADMIN` assignment. Custom-role create and update each write exactly one attributed audit row in the same transaction with canonical sorted before/after permissions; audit failure rolls back the role, and the bounded whole-transaction retry cannot duplicate the audit. The create controller maps only explicit domain conflicts plus recognized Prisma uniqueness and serialization/deadlock errors to 409. Raw audit/database failures propagate to the production filter's sanitized 500 response, while explicit service-unavailable exceptions retain sanitized 503 classification. Legacy control-bearing role names are rendered as bounded single-line labels without changing their IDs.

Email-addressed user creation, reactivation/delegation, RBAC assignment, encrypted delivery intent, and audit logging share one Serializable tenant transaction. Before any of those writes, the transaction locks and revalidates the live actor, exact request session, current permissions, delegation subset, selected role, and any archived target; actor revocation/suspension or protected-target denial therefore leaves user, assignment, outbox, and audit state unchanged. One bounded whole-transaction retry handles only recognized serialization/deadlock conflicts. Disabled delivery, invalid configuration, or any outbox encryption/insert failure rolls back the user. Username/PIN invitations create no email intent and return `invitationDelivery.status = not_applicable`; generated temporary PINs and username collision suffixes use the platform cryptographic random-number generator. Status, retry, and reissue require `users:admin`, remain tenant-scoped, and expose no recipient/template/token. Archived-user reactivation rotates any terminal delivery row to a fresh outbox/provider action ID, while retry preserves a nonterminal outbox ID so response-loss retries keep one provider identity. Reissue is allowed only from `DEAD_LETTERED`, requires a bounded `Idempotency-Key`, compare-and-set rotates the existing unique row to a deterministic fresh outbox ID and AES-GCM envelope, and writes the prior terminal ID/state plus key hash to immutable audit history in the same transaction. Same-key response-loss replay returns the current action; once that action becomes historical the key conflicts and a newly dead-lettered action requires a new key. Invitations are account access, not a scheduling solve, so creation, delivery, retry, and reissue never charge credits or create another user. Production requires canonical `STAFF_INVITATION_OUTBOX_ENABLED=true`, a dedicated exact 32-byte `STAFF_INVITATION_OUTBOX_ENCRYPTION_KEY`, and the same `STAFF_INVITATION_MAX_ATTEMPTS` value in API and worker; the database schema caps attempts at eight.

User deletion preserves the stable user ID for historical schedule and time-card references while replacing the required display name with a generic tombstone. The controller has no authorization preflight dependency: its Serializable tenant transaction locks and re-reads the live actor, target, assignments, roles, and permission edges in canonical order, rejects inactive or suspended actors, then performs authorization, hierarchy comparison, anonymization, credential/session cleanup, and attributable audit atomically. Suspended targets remain administrable for recovery, but a suspended actor can never delete another account. Under the Tenant-then-scheduling-advisory lock and ordered schedule-row locks, that transaction unassigns only active shifts on active draft schedules, including reopened schedules, plus active schedule-less shifts. Every distinct draft with an actual cleared assignment advances once, even when several target shifts share it; drafts with no changed assignment stay unchanged. Revision overflow or an incomplete assignment/revision/tombstone update aborts deletion. Published, archived, soft-deleted shift history, and every time-card user/shift reference remain unchanged. The transaction also clears plaintext and encrypted PII, lookup hashes, email-delivery suppression state, OIDC linkage, password/PIN/MFA material, role assignments, reset tokens and envelopes, TOTP replay claims, signup recovery attempts, notifications and their outbox rows, and availability-import requester links. Target availability-import jobs are row-locked and cleared of parsed results, encrypted PDF envelopes, and local storage references before the user tombstone can commit; completed imports remain charged, while undelivered imports are cancelled and exactly refunded. Source-file unlink is retried outside the transaction. Sessions are revoked, consumed refresh-validator hashes are destroyed, and current selector, validator, IP, and user-agent values are irreversibly tombstoned. Database triggers reject user deletion unless completed imports are erased and undelivered paid reservations are exactly refunded.

The admin PIN reset controller performs no target lookup, username write, hierarchy preflight, or pre-audit. AuthService validates a supplied PIN before writes, then owns one Serializable tenant transaction that locks actor and target users in stable order, locks and re-reads active assignment and permission edges, applies live `users:admin` plus strict hierarchy/subset authorization, optionally bootstraps an eligible local username, changes the PIN/reset flag, revokes every session, and writes the attributable redacted audit with bounded request IP/User-Agent. Self reset, same-rank reset, missing or concurrently revoked actor permission, concurrent target promotion, target permissions the actor does not hold, and a stale `SUPER_ADMIN` row without an active system assignment fail closed without partial reset state. Only a live user with both the `SUPER_ADMIN` legacy role and active system `SUPER_ADMIN` assignment receives the equal-rank override, and the controller returns the username committed by that transaction.

Email-only invitations require `auth:login_email`; `auth:login_password` alone is not accepted because invitations do not bootstrap a password. The default Staff role includes email OTP so invited staff can authenticate when OIDC is disabled.

The user directory uses validated, deterministic `createdAt`/ID cursor pagination with `limit + 1` and queries role assignments only for the returned page; it does not issue one RBAC query per user. Cursor payloads contain only the ordering timestamp and stable ID, internal hashes/timestamps are omitted from responses, and the tenant-leading active-user index supports the query. The initial page also includes exact active, manager, privileged, and PIN-account aggregate totals without loading or returning PII; cursor continuations skip the aggregate query.

Scheduling-profile reads require `users:read`; atomic replacements require `users:write`. Reads expose only active schedulable manager/staff accounts. Replacement locks the tenant and scheduling mutation boundary before revalidating the unsuspended tenant manager/staff row and every optional active tenant location, then invalidates affected drafts before deleting or creating skills and availability. Empty availability is returned explicitly and is passed to the solver as unavailable, not omitted as unrestricted.
