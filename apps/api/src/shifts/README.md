## Notes

Shift time updates lock dependent breaks and translate them by the shift-start delta. A resize that would move any translated break outside the new window returns a conflict before credit debit or mutation. Every locked break write is count-guarded; a miss, assigned-shift collision, or deferred shift/break database invariant returns 409 and rolls the shift, break, credit debit, and audit result back together. Create overlap validation remains a 400 request error.

Shift reads default to 100 rows, accept 1 through 200, and return pagination metadata with an opaque cursor in stable ascending `(startTime, id)` order. Optional UTC overlap windows and location/schedule filters preserve calendar workflows. Staff roster reads use the same limits and metadata with stable ascending `id` continuation; clients follow every page and sort the complete roster for display. A tenant/deletion/start-time composite index covers the broad tenant timeline while existing location, schedule, and user composites cover scoped calendar reads.

Shift reads include linked staff and break records for calendar consumers. Active reads exclude shifts whose location was soft-deleted; retained historical rows remain available only to audit and controlled export paths. Scheduler-facing reads and rosters only include open shifts, managers, and staff; admins and super admins are excluded from planner/lunch-break assignment surfaces. Staff reads recognize current RBAC role labels and self-scope to assigned shifts on published schedules. Create and assignment paths lock and revalidate active locations before mutation; billable mutations acquire tenant scheduling and schedule-row locks, then enforce paid-subscription-plus-configured-credit entitlement inside the write transaction so entitlement changes, location lifecycle, weekly-hour publish validation, and writes cannot race. Every committed scheduled create, update, delete, or changed bulk assignment increments each owning draft `Schedule.revision` in that same Tenant-then-advisory transaction. This invalidates any earlier publish preflight before settlement; replay and semantic no-op paths return before the revision write. Manual shift creation, update, and bulk assignment require an `Idempotency-Key`; actual mutations use a tenant-scoped hashed operation identity and canonical request hash retained with the original response in the immutable audit stream. Actual create, update, and bulk value changes debit the authoritative positive scheduling credit cost exactly once through the billing transaction before commit, alongside the domain mutation and audit record. Update payloads that are semantically equal to the locked shift return the current serialized shift without entitlement enforcement, credit ledger/debit, shift/break write, or audit/idempotency mutation. Bulk assignment likewise compares every requested value with the authoritative post-lock `userId`: an all-no-op batch returns `{ updated: 0 }` without entitlement, debit, row write, audit, or key reservation, while a mixed batch validates the complete requested state, charges once, and writes only changed rows. Because a no-op does not reserve its key, retrying that key re-evaluates the request against current state; once a real value change commits, same-key retries return the durable stored response without charging again, including after entitlement loss or a lost response. Request drift conflicts, and guarded-write failure rolls wallet, domain, revision, and audit work back together. Published and archived schedules are immutable. Delete remains an active-paid scheduling-entitled, non-billable correction available at zero wallet balance; it creates no ledger row. Shift timestamps remain UTC ISO 8601 instants; auto-created draft schedules use local calendar day boundaries. Draft mutations do not notify staff; schedule publish is the notification boundary.

When manual shift creation omits `scheduleId`, the API reuses a tenant- and location-scoped `DRAFT` schedule containing the full `[startTime,endTime)` interval. Only when none exists may it create a local-calendar fallback spanning the complete interval; any overlapping published or too-short draft schedule blocks fallback creation.
