## Notes

Lunch-break lists default to 100 rows, accept 1 through 200, validate optional UTC overlap windows, and return pagination metadata with an opaque cursor in stable ascending `(startTime, id)` order. Dashboard and day-planner clients provide explicit windows and follow every continuation page.

Shared schedule reads and persisted lunch/break actions run through `TenantPrismaService.withTenant` so PostgreSQL RLS receives the current tenant context. Every shared scheduling mutation acquires the `Tenant` row before the tenant scheduling advisory lock, matching normal shift writes and preventing setup/shift lock inversion. Policy reads, lunch/break lists, and policy updates are zero-settlement operations: they require active paid entitlement but remain available at zero wallet balance and never create a ledger row. Manual shift-break replacement still acquires the paid-subscription-plus-positive-credit gate inside its exact write transaction. Setup-shift persistence uses the authoritative scheduling entitlement and shared tenant scheduling lock because it creates or updates Shift rows. Setup batches require both `lunch_breaks:write` and `shifts:write`, a bounded `Idempotency-Key`, and an explicit tenant-owned location. Their canonical tenant/request identity, one operation-level scheduling debit, domain rows, owning draft revision increments for changed scheduled shifts, and immutable audit response commit atomically; replay remains available after entitlement loss, request drift conflicts, and any row failure rolls the debit and revision back. New unassigned rows also receive a tenant-locked semantic replay marker based on the normalized location/ordered-row request, with omitted and explicit-null users treated alike. That marker is checked after the scheduling lock and commits with the shift, separate-credit ledger, key audit, and response, so concurrent tabs or storage-loss retries using different keys return the original IDs without another creation or debit. Manual replacement also requires a bounded key. After draft-schedule locking and semantic comparison, an actual value change atomically debits the exact configured `lunch_breaks` cost, replaces breaks, increments the schedule revision, and records the deterministic audit response. Lost-response replay returns that response without another debit or write; same-key request drift returns `SHIFT_BREAKS_CONFLICT` with HTTP 409. An unchanged replacement returns the current row without entitlement lookup, debit, domain/revision/audit write, or idempotency reservation, while a distinct change without sufficient credits returns `SHIFT_BREAKS_ENTITLEMENT_REQUIRED` with HTTP 403 before mutation. Individual shift edits and persisted generation also require an explicit location; generation rejects shifts from any other location before the idempotency claim, credit charge, or schedule mutation. The API never guesses the tenant's first location. Preview generation remains standalone and does not require a location. Staff reads recognize current RBAC role labels, self-scope to the authenticated employee, and require a published schedule; draft break plans remain manager-only. Generation requires an `Idempotency-Key`; a tenant/key uniqueness boundary stores the canonical request hash, positive credit reservation, and successful response or classified failure so response-loss retries never charge or mutate twice. HTTP 403 records represent a current entitlement failure and 5xx records represent retryable persistence/service failure; an unchanged tenant/key/body may atomically reclaim either after subscription, separately purchased credits, or persistence health is restored. Only one claimant can move that row back to `PENDING`, and the original request ID keeps the debit identity stable. Terminal validation/conflict failures remain deterministic and do not re-execute. New completion requires billing to resolve to an active paid subscription with a safe positive separately purchased credit cost; plan, Stripe, manual, missing-cost, and zero-cost resolutions fail without a debit or reusable success. An already-debited successful request remains replayable after later entitlement loss. Credits are reserved before schedule mutation; a persistence exception rolls the whole transaction back before the classified failure is stored separately. Break replacement and the reusable success outcome commit atomically after schedule rows are locked and re-checked as drafts. Admins and super admins remain excluded from assignment surfaces. Persisted break rows store optional `break1`, `lunch`, or `break2` identity, while older untyped rows remain compatible.

Ownership follow-up: the service remains oversized because policy, placement, persistence, billing, and idempotency still share one class. A later bounded refactor should extract typed generation-workflow and break-placement collaborators while retaining durable replay and credit-compensation tests.
