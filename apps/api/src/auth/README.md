## Notes

Auth operational logs never serialize provider or Redis exception messages or stacks. The shared classifier emits only an allowlisted event, error class, category, known infrastructure code, and a strictly validated correlation ID. Non-production `AUTH_DEBUG` uses a separate runtime detail allowlist; submitted OTPs, tokens, credentials, raw redirect URLs, and arbitrary error details are not accepted.

Migrated legacy users authenticate with `username` plus preserved PHP `password_hash` values. PIN login remains available for accounts without a migrated password hash. Password reset requests return a generic response and atomically store a SHA-256 token hash plus an encrypted delivery outbox row. The worker leases due rows, sends with provider idempotency, retries transient failures to a fixed bound, and exposes dead-letter metrics/log alerts without logging recipients or reset tokens. Confirmation validates the token, account, tenant, and expiry before asynchronously hashing the new password; transactional compare-and-set consumption keeps each token one-time and revokes existing sessions.

Session cookie writers honor `COOKIE_SECURE`. The private HTTP dev deployment sets `COOKIE_SECURE=false` so browsers accept `access_token`, `refresh_token`, and `csrf_token`; HTTPS production should keep secure cookies enabled.

Cancelled workspaces without a deletion marker may authenticate so an administrator can reach billing settings and resubscribe. Suspended, purged, and deleted workspaces remain blocked. A user with `suspendedAt` set is excluded from login, password reset, refresh, MFA, session validation, and active-seat capacity until an authorized activation clears only that reversible lifecycle field. Billing entitlement resolution forces cancelled and delinquent workspaces to free-tier features and capacity until a new verified Stripe subscription becomes active.

Cookie-authenticated refresh requests require a matching `x-csrf-token` header when the browser-readable `csrf_token` cookie is present. Refresh uses an independent five-attempt SHA-256 credential bucket plus a 100-attempt source-IP ceiling per 15 minutes, so users behind one office NAT do not share the login-attempt budget and raw refresh credentials never enter throttler keys. Production public auth POSTs, including login flow resolution, require a same-origin `Origin` or `Referer` header before account-state lookups or cookie issuance. MFA enrollment, confirmation, disable, and verification routes are in the auth-attempt throttling bucket. Privileged MFA sensitivity is declared once on the canonical permission catalog and derived for login and refresh. It includes platform/admin, user/role write, billing/settings write, tenant lifecycle/export, `time_cards:approve`, and payroll policy-write, lock, export, and reconcile privileges; custom payroll-only roles therefore receive the same MFA gate. Unverified sessions may only read or complete MFA enrollment, verify MFA, refresh, logout, or reach `/auth/me`.

An admin-issued temporary PIN creates a reset-only session. Until `PUT /api/v1/users/me/pin` replaces that PIN, the JWT guard permits only self PIN rotation, refresh, logout, and `/auth/me`; it denies application and MFA enrollment routes. Admin reset validates a supplied PIN before opening the transaction, then locks actor and target users in stable ID order plus their assignments, roles, and role-permission edges. A suspended actor is rejected after this locked live read, while an active authorized actor may reset a suspended target for recovery without unsuspending it. Live permission/hierarchy authorization, target lookup, optional username bootstrap for local username accounts, PIN/reset mutation, every-session revocation, and attributable IP/User-Agent-bounded redacted audit share one Serializable tenant transaction. Username generation checks active and retained soft-deleted rows, and same-name reservation conflicts receive a bounded whole-transaction retry before returning a controlled conflict. A concurrent actor revocation or target promotion therefore denies or serialization-fails closed without a partial username, credential, session, or PIN-audit change. Self-rotation preserves its existing PIN/MFA semantics. Redis MFA-marker cleanup is post-commit and best-effort because the revoked database session is authoritative. After rotation, a new login and refresh re-evaluate live MFA state, so an already-enrolled account must still verify its existing authenticator or recovery code before application access.

`RbacGuard` fails closed for authenticated routes. Routes must be `@Public()`, `@RequirePermission(...)`, or explicitly `@AllowAuthenticated()` when only a valid session is required. Multiple permissions in route metadata use all-of semantics. Guarded requests replace JWT legacy-role claims with the current `User.role` value loaded during session validation. User role/access mutations additionally ignore request role arrays for authority, lock and re-read live database state, reconcile legacy and assigned roles, and revoke active sessions whenever effective access changes. Protected or system-admin delegation requires both current `User.role = SUPER_ADMIN` and an active, nondeleted system assignment with `legacyRole = SUPER_ADMIN`; a stale row or assignment alone never bypasses subset checks. Custom role writes apply the same live dual-source requirement in their mutation transactions.

Payroll authorization is split into `time_cards:approve`, `payroll:read`, `payroll:policy_write`, `payroll:lock`, `payroll:export`, and `payroll:reconcile`. All 16 PayrollController routes are part of the explicit authorization inventory with exact metadata assertions. System Super Admin and Admin roles receive all six; Manager receives only time-card approval and payroll read access; Staff keeps its existing clock permissions without any payroll-control grant. Tenant-defined roles receive only the permissions explicitly selected for that role.

Custom roles can be deleted only when their tenant-scoped assignment count is zero. Deletion carries the actor identity, reauthorizes it from locked live RBAC state, records an attributable audit, and maps only recognized Serializable database conflicts. The transaction rejects assigned-role deletion with a conflict and leaves both the role and assignments unchanged.

Role names reject C0/C1 control characters before create or update database work. Existing legacy names are normalized to a bounded single-line display value on reads, while immutable role IDs remain unchanged. User access replacement deduplicates role IDs and permits at most 100 active assignments, matching the web identity-header parser's exact bound.

OIDC token and userinfo requests use the SSRF-hardened secure HTTP client with fixed eight-second deadlines, disabled redirects, bounded response bodies, and JSON-object validation. Provider timeouts, oversized bodies, malformed JSON, and invalid token shapes fail closed as fixed unauthorized responses without logging tokens or provider bodies. OIDC login requires `email_verified: true`, a valid provider subject, and an existing user in the selected workspace. The first successful login atomically binds the configured issuer and provider subject to that user; later issuer/subject or email mismatches are rejected instead of rematching by email. Issuer/subject uniqueness is tenant-scoped, allowing one provider identity to use separately invited accounts in multiple workspaces without permitting duplicate bindings inside one workspace. Each OIDC state is also bound to the initiating browser by a separate host-only HttpOnly `SameSite=Lax` correlation cookie. Redis stores only the nonce hash, and the callback clears the cookie before accepting or rejecting the state.

Pre-auth login resolution, password, PIN, email OTP, and password-reset routes use separate per-endpoint layered buckets. The five-attempt bucket combines the trusted Express client IP with a normalized, SHA-256-hashed tenant/identifier (or reset token), so one source cannot spend another NAT user's budget and traffic from another IP cannot exhaust a victim's account bucket. An independent thirty-attempt source-IP ceiling limits identifier spraying. Both layers use only `req.ip`, which follows the configured trusted-proxy contract, and canonicalize IPv4, IPv4-mapped IPv6, and IPv6 forms. Migrated password and PIN verification serialize account-state decisions with a database row lock so parallel failures cannot overwrite the threshold lock. Missing credentials and active lockouts perform dummy KDF work and return the same unauthorized contract; SSO-only policy is checked before identity lookup. Throttle hashes remain only in Nest's process-local TTL store and expire after 15 minutes; login, reset, refresh, and MFA throttles do not create relational identifier records. OTP challenge, send-limit, failed-attempt, and lock keys expire from Redis within their one- or ten-minute policy windows. Redis stores only an `OTP_HMAC_SECRET`-keyed digest of each live code; production requires a dedicated secret and plaintext exists only transiently for delivery.

`POST /api/v1/admin/retention/purge-expired` may authenticate with `RETENTION_PURGE_SERVICE_TOKEN` or `RETENTION_PURGE_SERVICE_TOKEN_FILE`. That identity is accepted only on this route, receives only `admin_portal:access`, may execute only confirmed 30-day `application_data` purges, and may only dry-run seven-year `retained_records` discovery. Final retained-record deletion still requires a platform-admin identity. The same daily application-data run executes fixed 24-hour onboarding signup-attempt retention: active OTP, lockout, retry, and recovery rows remain available, while expired stable identity and organization hashes are deleted through a platform-capability-gated database function. Dry-runs report eligible rows without mutation.

Production email OTP requires `RESEND_API_KEY` and `EMAIL_FROM`. `EMAIL_OTP_DELIVERY_DEADLINE_MS` defaults to 10000 and must remain between 100 and 30000 milliseconds; one deadline covers the suppression lookup and provider handoff so an auth request cannot wait indefinitely. Local development can omit `RESEND_API_KEY`; delivery then fails closed instead of logging OTPs or recipient data. Public onboarding OTPs are scoped to the submitted organization name before tenant provisioning. Verification allows at most five failed attempts per workspace/onboarding challenge and email before invalidating and locking it for ten minutes, independent of source IP. Failed-attempt counters survive code reissuance, active locks suppress generation with the same generic response as the send throttle, and a fresh code is available only after the lock and counter TTLs expire. Successful verification clears the challenge protection state. Provider success, suppression, and failure logs omit recipient addresses and OTP values; failures use only the bounded auth operational diagnostic.

Tenant email OTP requests dispatch only for an active, non-deleted user whose effective permissions include `auth:login_email` and whose workspace permits email login. Unknown, deleted, blocked, and otherwise ineligible recipients receive the same generic success response without OTP generation or an email-provider call.

New sessions issue one opaque versioned cookie containing a stable selector and rotating validator. Session creation locks and rechecks the tenant before locking the user, then removes expired and revoked credentials and retains at most 20 active sessions per user. This tenant-to-user order matches lifecycle mutation ordering, so a committed suspension cannot be followed by a newly issued unrevoked credential, and later activation does not revive a captured failed login. The database stores only independent SHA-256 hashes in `Session.selectorHash` and `Session.refreshToken`; refresh rotates only the validator, while logout revokes by the stable selector so concurrent rotation cannot preserve the session. Each selected refresh locks the selector row and stores the consumed validator hash in `RefreshTokenReplay` before rotating. A later selector-matched validator revokes the session only when that validator appears in the consumed ledger, so stale-token replay terminalizes the winning family while arbitrary selectors or validators cannot force logout. Existing single-part plaintext or hashed refresh rows remain accepted and are upgraded to selector-based credentials on refresh. The consumed legacy credential hash is written to the replay ledger before that upgrade, so concurrent or later predecessor replay revokes the upgraded family.

Production MFA enrollment requires the exact 32-byte hex/base64 `MFA_SECRET_ENCRYPTION_KEY_CURRENT`; new TOTP secrets are stored as AES-GCM `enc:v2` payloads containing a non-secret key reference. `MFA_SECRET_ENCRYPTION_KEY_PREVIOUS` supplies bounded managed-key overlap, while deprecated `MFA_SECRET_ENCRYPTION_KEY` reads legacy `enc:v1` rows until transactional rotation finishes. Plaintext rows remain readable only for migration compatibility. Malformed envelopes, missing referenced keys, duplicate managed keys, and bad key lengths fail closed. One-time backup codes remain scrypt-hashed. Enrollment confirmation prepares its access token before the database transaction can commit. After commit, pending-enrollment deletion and session-marker creation are independent best-effort Redis writes with sanitized diagnostics; either or both may fail without replaying the committed transaction or withholding the only plaintext copy of the backup codes. Backup-code verification locks the tenant user row and validates and removes the matching hash in the same database transaction. MFA disable removes factors, revokes every active session, and records the audit in that same transaction; Redis marker deletion is only best-effort cleanup and cannot keep a revoked token authorized. Successful TOTP challenges atomically create a tenant-scoped `MfaTotpClaim` for the accepted user/time-step; the unique database constraint permits only one session to claim that step, and storage errors fail closed. Only the session whose transaction consumes the code is marked MFA-verified, so concurrent sessions cannot reuse one recovery code. Follow `docs/runbooks/mfa-encryption-key-rotation.md` before removing an overlap key.

Password reset delivery uses a durable encrypted outbox. `APP_ORIGIN` must be the browser-visible public HTTPS origin in production; API startup and Compose fail closed when it is blank or omitted, and runtime fallback values are accepted only after the same public-origin validation. `PASSWORD_RESET_OUTBOX_ENCRYPTION_KEY` must be a dedicated 32-byte hex or base64 secret shared by the API and worker; production launch validation and Compose fail closed when it is absent.

Public workspace provisioning is controlled by `PUBLIC_SIGNUP_MODE`. Missing production config fails closed as `closed_beta`; set `open` only when public self-serve tenant creation is intentionally enabled, or `invite_only` with `PUBLIC_SIGNUP_INVITE_CODES` for gated onboarding. A newly provisioned workspace receives a 14-day Starter trial with zero credits. The trial supports onboarding and account setup only; every billable feature remains disabled until both a paid subscription and separately purchased or administratively granted credits exist. Turnstile verification uses the same secure HTTP client with an eight-second deadline, a 16 KiB response cap, disabled redirects, and JSON-object validation; provider failures return a fixed unavailable response while rejected challenges remain forbidden. Production `open` signup requires Cloudflare Turnstile: set `TURNSTILE_SECRET_KEY` for the API and `NEXT_PUBLIC_TURNSTILE_SITE_KEY` for the web client, and submit the token as `turnstileToken` with onboarding OTP requests. The web signup UI also reads `NEXT_PUBLIC_SIGNUP_MODE`; keep it aligned with the server mode.

The API and web import `PUBLIC_LEGAL_MANIFEST` through the `@lunchlineup/config` package export so self-service gating, rendered document versions, and recorded assent cannot drift. Legal version changes belong only in that immutable manifest.

Public provisioning requires separate Terms and Privacy assent flags. The tenant, owner, `PUBLIC_SIGNUP_LEGAL_ASSENT` audit evidence, default RBAC roles, and owner role assignment are created in one platform-admin transaction; an RBAC failure rolls back the workspace instead of leaving an orphan. The evidence records server-owned legal versions, timestamp, owner user/email, IP, and User-Agent. The verify response returns the generated workspace slug so the client can retain it without persisting the OTP.
