# Prometheus

## Files

- `README.md`: this Prometheus configuration guide.
- `promtool-validation-credentials.txt`: checked-in non-secret placeholder mounted only while promtool validates credential-file references.
- `prometheus.yml`: scrape configuration for Prometheus, API, engine, worker, webhook replay, control-plane, and node-exporter metrics.

## Folders

- `alerts/`: production alert groups loaded by Compose plus an isolated `tests/` fixture subdirectory.

## Notes

The API metrics scrape uses the Docker secret mounted at `/run/secrets/metrics_token`. `promtool-validation-credentials.txt` is deliberately not a runtime secret; the verifier mounts it at that path only so pinned-image config validation can open the declared credential file without exposing production credentials. Compose mounts `alerts/` read-only at `/etc/prometheus/alerts`; the non-recursive `/etc/prometheus/alerts/*.yml` runtime glob sees only the two root production group files and cannot load promtool fixture syntax from `alerts/tests/`. Targets use Compose service DNS names (`api`, `engine`, `worker`, `webhook-replay`, `control`, and `node-exporter`) so multiple stacks can run with different `COMPOSE_PROJECT_NAME` values without fixed-container-name collisions. The Compose config uses literal external labels because `prom/prometheus:v2.51.2` does not support `--config.expand-env`.

Every alert in `alerts/lunchlineup.yml` must include a `runbook` annotation pointing at an existing `docs/runbooks/*.md` file. API scrapes run the bounded dependency checks before rendering metrics. `RequiredApiDependencyUnavailable` fires when the database, Redis, or RabbitMQ dependency metric is zero or absent, so a reachable API process cannot produce a false-green required-dependency state. Paired fast- and slow-burn alerts enforce the 99.9% API and public-web availability objectives documented in `docs/runbooks/service-level-objectives.md`. Production readiness also requires backup freshness telemetry through `lunchlineup_backup_last_success_timestamp_seconds`, stage-and-mode-scoped retention telemetry through `lunchlineup_retention_purge_last_attempt_timestamp_seconds`, and public edge telemetry through `lunchlineup_public_web_probe_success` plus `lunchlineup_public_web_probe_last_attempt_timestamp_seconds`; absence of these metrics is a critical alert, not a silent gap. Solver backlog sums worker-owned ready and retry broker gauges. `SolverQueuePoisoned` fires immediately on any durable DLQ depth or recent terminal transition, so one indefinite poison item cannot hide below the backlog threshold. Notification, password-reset, and staff-invitation dead-letter alerts use 15-minute counter increases rather than lifetime terminal-row gauges, so they fire on a recent incident and resolve after no new terminal transition while database/audit history remains available. Provider/readiness/freshness alerts route to `docs/runbooks/outbound-delivery.md`; invitation staleness compares the last healthy cycle against the worker-exported configured maximum instead of a second hard-coded threshold.

`alerts/tenant-deletion-billing.yml` uses only `lunchlineup_tenant_deletion_billing_*` series, keeping it distinct from customer-cancellation reconciliation. `TenantDeletionBillingBacklogSustained` warns only when backlog is nonzero and its oldest barrier exceeds 15 minutes for another 10 minutes. `TenantDeletionBillingSweepStale` pages when the last successful sweep is absent or older than the API-exported maximum staleness, even while failed sweeps continue updating `last_sweep`. Both route to `docs/runbooks/data-retention-delete-export.md`.
`PdfParserUnavailable` pages after two minutes of worker-observed parser loss; `PdfParserReadinessMissing` pages when the worker scrape cannot prove parser readiness. Both route to `docs/runbooks/production-readiness.md` and use the worker-side `lunchlineup_pdf_parser_ready` gauge so parser failure after worker startup is observable.

`infrastructure/control/public-web-probe.sh` runs from the host every minute and writes `lunchlineup_public_web.prom` to the same node-exporter textfile collector. `PublicWebUnavailable` pages after bounded HTTPS failures, including timeouts and Caddy `502` responses. `PublicWebProbeStale` pages when the timer or metric path stops reporting. The probe verifies the canonical public HTTPS root, deployed release header, and rendered Next.js page without granting any Compose service new egress.

The Compose `backup` service writes `lunchlineup_backup_last_success_timestamp_seconds` and `lunchlineup_backup_last_success_size_bytes` to `/metrics/lunchlineup_backup.prom`. Its writable `/metrics` bind and node-exporter's read-only `/textfile_collector` bind resolve from the same `${NODE_EXPORTER_TEXTFILE_DIR:-/var/lib/node_exporter/textfile_collector}` host path. `lunchlineup-backup.service` must therefore produce `/var/lib/node_exporter/textfile_collector/lunchlineup_backup.prom` on its first one-shot run before the timer is enabled. Production Terraform requires `backup_metrics_collector` to be either `node-exporter-textfile:/absolute/path/lunchlineup_backup.prom` or `authenticated-metrics:https://...`; do not launch with backup freshness only written to a local file that Prometheus cannot scrape.

Platform on-call owns `BackupMissingTelemetry`, `BackupStale`, `PitrBaseBackupTelemetryMissing`, `PitrBaseBackupStale`, `PitrWalArchiveFailure`, and `PitrWalArchiveStale`. The PITR base-backup job writes `lunchlineup_pitr.prom`; PostgreSQL's archive command writes `lunchlineup_pitr_wal.prom` after each remotely verified success or failed attempt. Validate the relevant timer or archive command, exact remote object, textfile mtime/content, node-exporter scrape, and Prometheus rule state before resolving an alert. Use `docs/runbooks/postgres-pitr-recovery.md` for PITR alerts.

`scripts/invoke-retained-record-purge.mjs` writes stage-specific textfile metrics for both systemd schedules. Confirmed 30-day application-data execution writes `lunchlineup_retention_purge.prom` with `mode="execute",stage="application_data"`; seven-year candidate review writes `lunchlineup_retained_record_review.prom` with `mode="dry_run",stage="retained_records"`. `ApplicationDataRetentionExecutionTelemetryMissing` and `ApplicationDataRetentionExecutionStale` select only the automatic execution series and page independently of successful dry runs, with the stale boundary fixed at 26 hours. Compose mounts `${NODE_EXPORTER_TEXTFILE_DIR:-/var/lib/node_exporter/textfile_collector}` read-only into node-exporter and enables `--collector.textfile.directory=/textfile_collector`. `RetentionPurgeCandidatesReady` applies only to the dry-run series and opens an operator-review ticket; final retained-record execution remains a reviewed platform-admin action. Run `node scripts/verify-observability-configs.mjs --root . --tool-mode container` to validate the production config/rules and execute both checked-in promtool rule fixtures with the digest-pinned image.
