# PostgreSQL PITR

## Files

- `README.md`: this Postgres runtime and PITR configuration guide.
- `archive-wal.sh`: fail-closed `archive_command` target that rejects disabled PITR, accepts only exact PostgreSQL archive names, and returns success only after the private request-scoped provider returns conditional immutable upload plus exact versioned byte/retention proof.
- `pg_hba.conf.template`: generated client-authentication template.
- `pitr-object-store.sh`: shared archive-name classifier, strict stat-state handling, mode-0600 MinIO config channel that keeps credentials out of argv, validated timeout/kill-after wrapper, conditional encrypted upload, single-version enumeration, and version-specific readback helpers.
- `pitr-export-lifecycle-policy.sh`: exports the live bucket lifecycle JSON through the dedicated read-only lifecycle-audit identity for canonical policy verification.
- `pitr-verify-object-store.sh`: fail-closed writer preflight for bucket versioning, default COMPLIANCE retention, per-object retention, and denied deletion.
- `pitr-wal-provider-upload.sh`: provider-side WAL upload entrypoint that emits exact immutable version proof only after encrypted byte and retention readback succeeds.
- `postgresql.conf`: checked-in Compose Postgres runtime configuration with archiving off by default.
- `postgresql.conf.template`: generated Postgres tuning template with the same non-PITR-safe archive default.
- `restore-wal.sh`: `restore_command` target that requires exactly one immutable provider version and retrieves it with an explicit version ID.

## Contract

Production sets `PITR_ENABLED=true` and `PITR_ARCHIVE_MODE=on`, a unique cluster-specific `PITR_S3_PREFIX`, an HTTPS endpoint/bucket, and `PITR_OBJECT_LOCK_RETENTION_DAYS` of at least 14. Every non-PITR render leaves `archive_mode=off`; invoking `archive-wal.sh` without enabled PITR fails nonzero and never acknowledges an unarchived segment. The bucket must have versioning enabled and default COMPLIANCE Object Lock for that exact duration. Every upload also requests COMPLIANCE retention explicitly. Exact-prefix lifecycle rules must preserve current objects through that immutable minimum, expire every noncurrent version and delete marker, and keep current plus noncurrent expiry within `PITR_LIFECYCLE_MAX_RETENTION_DAYS` (no more than 90).

Credentials are read from the mounted secret files into a private mode-`0600` temporary `mc` config and are never passed in `mc` argv; only the protected config-directory path is visible to a running provider process. Every `mc` provider call uses GNU `timeout --signal=TERM` with validated bounds and KILL escalation inside a one-shot container ownership domain. The owner snapshots container processes, rejects surviving descendants with exit `70`, and relies on exact container teardown before retry. A timeout or authorization/transport failure is unknown state, not object absence. Uploads use `If-None-Match:*`; uncertain mutations reconcile through exact single-version authenticated stat/download staged beside the source rather than capacity-limited `/tmp`, while conflicting or unversioned history fails closed.

Postgres receives only the private `pitr-wal-provider` URL and never receives object-store credentials or provider tooling. The request-scoped WAL provider mounts only `PITR_WAL_OBJECT_STORE_SECRETS_DIR`; the daily base-backup job mounts only `PITR_BASE_BACKUP_OBJECT_STORE_SECRETS_DIR`; recovery mounts only `PITR_RESTORE_OBJECT_STORE_SECRETS_DIR`. WAL and base-backup identities may put, stat, read for idempotency verification, inspect bucket version/retention configuration, and inspect object retention, but must be denied every object/version delete action. The restore identity is read-only. No application or data container receives lifecycle credentials. The isolated `pitr-lifecycle-audit` job uses a fourth, distinct `PITR_LIFECYCLE_AUDIT_OBJECT_STORE_SECRETS_DIR` identity with configuration-read permission only; lifecycle administration and deletion remain external. A protected provider-specific authorization simulator is pinned by `PITR_AUTHORIZATION_SIMULATOR_FILE` and `PITR_AUTHORIZATION_SIMULATOR_SHA256`; it must authenticate with each identity and return fresh provider-API decisions for the exact bucket/prefix without issuing object or policy mutations.

`scripts/pitr-verify-storage.sh` actively uploads one immutable canary with each writer identity and requires the delete attempt to fail while the object remains readable. It requires fresh provider authorization simulations for both restore and lifecycle-audit identities, fails if any required read is denied or any write/delete/retention/lifecycle/policy/bucket mutation is allowed, then canonicalizes the live lifecycle export and compares it with the digest-pinned retained proof and declared maximum. Production deploy runs this before its mutation marker, and backup readiness repeats it before enabling timers. `archive_command` only returns success after encrypted remote durability proof; PostgreSQL therefore retains/retries a segment when off-host storage is unavailable.

Archive expiry belongs to a separately managed bucket lifecycle policy and identity outside this Compose stack. That identity must never be mounted in Postgres, backup, restore, API, worker, or migration containers. Logical encrypted dumps remain a separate recovery path and are not replaced by PITR.
